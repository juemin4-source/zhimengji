# 织梦机 Zhimengji v1.2 — 产品规格

## 版本概述

| 字段 | 值 |
|------|-----|
| 版本号 | v1.2 |
| 产品名 | 织梦机 Zhimengji |
| 类型 | 智能世界构建与叙事引擎 |
| 技术栈 | Tauri + React + TipTap + @xyflow/react |
| 当前状态 | v0.2 — 功能可用但存在大量 UX 问题 |
| 设计季节 | **春生** — 找 PMF、稳体验、赢信任 |
| A 星 | **金（刀子）** — 面向创作者的规划与写作工具 |
| B 星 | **金（刀子）** — 买断制/订阅制专业工具 |
| 造局检查 | A(金) + B(金) = **同局**。没有转化损耗，工具即价值。警惕单一脆弱（缺乏网络效应），需维持产品深度。 |

### A/B 星分析

织梦机的入口（A 星）是 **金（刀子）**：用户带着明确的任务来——组织世界设定、连接人物关系、梳理叙事结构。不是来社交、不是来刷内容，而是来"干活"。

织梦机的出口（B 星）也是 **金（刀子）**：用户为工具的专业性付费。买断制或订阅制，用户决策基于"这个工具能帮我省多少时间、理清多少混乱"。

**同局含义**：能量流向通顺。用户的付费动机就是使用动机。缺点是缺乏平台效应和网络效应，意味着织梦机必须靠产品深度和体验质量守住护城河，不能靠规模。

### 季节判断：春生

v1.2 处于 **春生** 季节。核心任务：

- **T（信任）**：存储和数据安全是红线。任何数据丢失场景必须在 P0 解决
- **η（留存/转化）**：当前的 UX 问题导致用户用完即走或不敢用。流畅的核心体验是留存关键
- **A（入口）**：让新用户从"下载"到"写出第一个设定"的路径变短

**春生不看 B（利润）**：v1.2 不追求变现。先让产品值得付费，再考虑定价。

## P0 — 必须修复（存储安全、数据丢失）

### P0-01：API 调用可靠性（fire-and-forget 问题）

**现状**：所有 `api.*` 调用是 fire-and-forget。`onUpdateObject` 中调用 `api.updateWorldObject` 只 `.catch` 弹 toast，失败的数据没有重试、没有队列、没有本地缓存。网络抖动一次，用户几分钟的编辑就丢了。

**产品定义**：

- 引入 **本地写入优先 + 后台同步队列** 模式
- 每一次编辑先写入本地状态（已实现乐观更新），同时加入同步队列
- 队列自动重试（3 次，指数退避）
- 队列持久化到 IndexedDB 或 SQLite，应用重启后恢复
- 同步失败在底部状态栏显示"待同步"计数

**验收标准**：
- 断网后编辑不丢失，恢复网络后自动同步
- 连续 3 次重试失败后显示持久化错误提示，提供手动重试按钮
- 重启应用后未同步的更改不会丢失

**处理方式**：做

---

### P0-02：离线检测与处理

**现状**：无任何网络状态感知。Tauri 后端如果不可用（进程崩溃、SQLite 锁、路径问题），前端完全不知道，继续操作的结果静默丢失。

**产品定义**：

- 前端定期 ping Tauri backend health endpoint（如 `ping` command）
- 后端不可用时前端自动进入 **离线模式**：
  - 编辑操作正常进行（写入前端状态）
  - 所有变更入同步队列
  - 界面顶部显示黄色横幅："数据暂存本地，正在尝试连接…"
- 后端恢复后自动回放队列

**验收标准**：
- backend 停止后 3 秒内前端进入离线模式
- 离线期间编辑不丢失
- backend 恢复后 5 秒内自动同步

**处理方式**：做

---

### P0-03：版本历史与全局撤销

**现状**：编辑器有 TipTap undo/redo（只在当前编辑 session），但画板无撤销。没有全局版本历史。用户做了破坏性操作（如删除对象、清空画板）后无法恢复。

**产品定义**：

- 在 `App.tsx` 层维护一个轻量级变更日志（`Changelog`），记录每次关键操作的快照或增量差异
- 支持撤销栈：对象创建/删除、画板位置变更、连接创建/删除
- 每次 session 开始时自动创建一个命名快照点
- 高级：按时间维度查看对象在不同版本的状态（v1.2 做基础版，不做完整时间旅行）

**验收标准**：
- 删除对象后可通过撤销恢复
- 画板拖拽误操作可通过撤销回退到之前位置
- 关闭应用后重新打开，撤销历史不保留（v1.2 特性边界）

**处理方式**：做（基础版）

---

### P0-04：画板状态保存同步

**现状**：`saveCanvasState` 在 `onUpdateCanvasState` 中被调用，但 `setCanvasStates` 是异步批处理，而 `api.saveCanvasTabState` 在前一次可能未完成时就再次触发。拖拽操作频繁触发保存，可能导致覆盖。

**产品定义**：

- 引入去抖（debounce）机制：画板变化后 500ms 内未再有新变化才触发保存
- 引入版本戳：每次状态快照携带自增版本号，后端保存时校验——如果后端的版本号比当前新，说明有其他 session 先写了，弹出冲突提示
- 拖拽结束时只保存一次，而非每像素移动都保存

**验收标准**：
- 连续拖拽 30 秒松开后，只触发 1-2 次保存，而非数百次
- 版本戳冲突有明确提示

**处理方式**：做

---

### P0-05：数据导入/导出

**现状**：没有数据的导入导出能力。用户的数据完全锁定在织梦机的 SQLite 数据库中。没有迁移路径，没有备份手段。

**产品定义**：

- 导出为 `.zhimengji` 包（zip 格式，内含 project.json + markdown 文件）
- 导入支持：`.zhimengji` 包、纯 Markdown 目录结构
- 导出时可选择导出范围：单个项目 / 全部项目
- 在书架界面添加"导出/导入"入口

**验收标准**：
- 导出内容包含项目的所有对象、连接、画板状态
- 导出后的 `.zhimengji` 文件可重新导入创建一个完全一致的项目
- Markdown 导入可以识别 `# 标题` 为对象名称，`[[WikiLink]]` 为引用

**处理方式**：做

## P1 — 本版做（核心体验）

### P1-01：创作向导（新建作品流程改进）

**现状**：点击"新建作品"直接创建名为"新作品"的项目并进入编辑器。没有体裁选择、没有项目描述、没有模板选择。

**产品定义**：

- 将"新建作品"改为两步向导：
  - 第一步：输入作品标题、选择体裁（科幻/奇幻/武侠/现实/悬疑/自定义）
  - 第二步（可选）：选择初始场景模板（"从零开始" / "经典三幕式" / "角色驱动" / "世界先行"）
  - 基于体裁和模板自动预填充初始标签和推荐对象类型
- 保留一键"快速创建"入口（跳过向导，使用上次选择或默认配置）

**验收标准**：
- 新用户可以在 3 次点击内完成创作开始（含标题输入）
- 体裁选择影响后续编辑器的初始配置

**处理方式**：做

---

### P1-02：首次启动引导

**现状**：创建作品后直接进入空文档视图。用户面对空白界面不知从何开始。正典系统、画板、WikiLink 等核心概念没有任何引导。

**产品定义**：

- 首次创建项目后弹出轻量引导层（3 步，可跳过）：
  - 第一步："这里有 8 种对象类型——人物、地点、组织…试着创建一个角色"
  - 第二步："在文档中写入 [[对象名]] 可以创建双向链接"
  - 第三步："画板可以可视化你的世界结构"
- 引导只对每个项目触发一次
- 侧边栏底部增加"?"帮助按钮，随时可重新打开引导
- 空文档视图中显示"从模板创建"的显式引导卡片

**验收标准**：
- 首次使用的新用户在引导完成后，知道如何创建一个对象和建立链接
- 引导可跳过，跳过不影响功能

**处理方式**：做

---

### P1-03：文档格式——HTML vs Markdown

**现状**：TipTap 默认存储为 HTML，但用户的 `currentObject.content` 在源码模式直接暴露为字符串，预览模式下使用 `markdownToHtml` 转换，造成了格式选择的矛盾。用户期望用 Markdown 写作，但编辑器内部用 HTML。

**产品定义**：

- 采用 **Markdown 作为规范存储格式**，TipTap 作为 Markdown 的 WYSIWYG 渲染器
- 引入 `@tiptap/extension-markdown` 或自定义序列化器，在 `onUpdate` 时存储为 Markdown
- 源码模式下编辑的是原始的 Markdown（和存储一致）
- WYSIWYG 模式下修改即时转换为 Markdown 保存
- 预览模式下直接渲染 Markdown

**验收标准**：
- 切换到源码模式看到的是 Markdown 格式，而非 HTML
- 在源码模式编辑 Markdown 后切回 WYSIWYG，格式保留
- 现有用户的 HTML 内容自动迁移为 Markdown

**处理方式**：做

---

### P1-04：合并工具栏

**现状**：WYSIWYG 模式和 Source 模式各有独立的工具栏，功能重复（B/I/H2/H3/Blockquote/List/Code），但实现方式不同。用户在切换模式时会看到两组按钮，造成认知负担。

**产品定义**：

- 设计单一工具栏，根据当前编辑模式显示可用的按钮
- WYSIWYG 模式下活跃按钮高亮（如 `editor.isActive('bold')`）
- 源码模式下按钮插入 Markdown 语法标记
- 模式切换按钮（编辑/源码/预览）整合到工具栏右侧
- 移除模式切换时的 UI 闪烁

**验收标准**：
- 同一个工具栏在不同模式下功能一致（都是加粗，实现方式不同）
- 用户不需要学习两套工具栏

**处理方式**：做

---

### P1-05：保存状态指示器

**现状**：`api.updateWorldObject` 是 fire-and-forget，成功或失败只在 toast 闪现 2 秒。用户无法知道当前文档是否已保存、是否有待同步的更改。

**产品定义**：

- 在底部状态栏左侧添加保存状态指示器：
  - 绿色圆点 + "已保存" — 所有更改已同步到 backend
  - 黄色圆点 + "保存中…" — 有更改正在同步
  - 红色圆点 + "保存失败" — 同步失败，可点击查看详情
  - 灰色圆点 + "离线" — 无连接，更改暂存本地
- 编辑器内容变化标记 `contentDirty`，驱动指示器状态

**验收标准**：
- 用户始终能看到当前编辑的保存状态
- 保存失败时提供重试入口

**处理方式**：做

---

### P1-06：画板缩放控件

**现状**：`CanvasView` 有 `scale` 状态管理和 `panOffset` 计算，但没有可视化缩放控件。用户无法直观地放大/缩小画布。`handleAutoLayout` 触发时重置缩放，但用户没有控制权。

**产品定义**：

- 在画板右下角添加缩放控件面板：
  - 放大（+）/缩小（-）按钮，步进 0.1
  - 当前缩放百分比显示（如 "100%"）
  - "适应画布"按钮（现有 `auto-layout` 不变）
  - 缩放到 100% 按钮
- 缩放范围：0.2x — 3.0x
- 快捷键控制缩放：Ctrl+滚轮 / Ctrl++ / Ctrl+-

**验收标准**：
- 用户可以通过控件将画布缩放到任意比例
- Ctrl+滚轮缩放流畅
- "适应画布"确保所有节点可见

**处理方式**：做

---

### P1-07：正典系统引导

**现状**：正典等级（未收录/草案正典/项目正典/核心正典）是织梦机的核心概念，但没有任何新手引导。用户在 Inspector 中看到"收录为设定"按钮但不知其含义，设定集只显示"非未收录"对象但用户不知如何收录。

**产品定义**：

- 用户创建第三个对象后，弹出轻量提示："你已经创建了几个对象——可以尝试把它们收录为设定"
- 在 Inspector 的"收录为设定"按钮旁添加信息图标，hover 时展示正典等级说明：
  - "未收录 → 草案正典：初步确认的存在"
  - "草案正典 → 项目正典：团队认可的核心设定"
  - "项目正典 → 核心正典：不可动摇的世界基石"
- 在设定集顶部添加一个简短的说明行："这里显示正典等级 > 未收录的对象"

**验收标准**：
- 新用户在创建 5 个对象之前知道如何"收录为设定"
- 用户能说出 4 个正典等级的大致含义

**处理方式**：做

---

### P1-08：全局搜索

**现状**：没有任何全局搜索功能。用户只能在当前对象大纲中遍历查找，当对象数超过 20 个时效率极低。

**产品定义**：

- 在导航栏中添加全局搜索入口（快捷键：Ctrl+K 或 Cmd+K）
- 搜索范围：对象名称、内容、标签、别名
- 搜索结果分组显示（按对象类型分组）
- 搜索结果点击跳转到对应对象的文档视图
- 支持模糊搜索和拼音搜索（中文场景）

**验收标准**：
- 输入关键词后 300ms 内显示搜索结果
- 搜索结果包含所有相关对象
- 点击结果跳转到对象文档

**处理方式**：做

## P2 — 排期（重要但不紧急）

### P2-01：书架搜索/筛选/排序

**现状**：Bookshelf 只有简单的卡片网格展示，无搜索、无类型筛选、无排序选项。

**产品定义**：
- 书架顶部添加搜索框，按作品名称/体裁搜索
- 体裁筛选下拉框
- 排序选项：最近更新、创建时间、作品名称、字数
- 列表/网格视图切换

**处理方式**：做

---

### P2-02：作品编辑/删除

**现状**：作品一旦创建无法修改标题、体裁，也无法删除。

**产品定义**：
- 书架卡片右上角添加 "..." 菜单（编辑/删除）
- 编辑对话框：修改标题、体裁、作品状态（构思中/草稿中/修改中/已完成）
- 删除确认对话框
- 删除后自动刷新书架

**处理方式**：做

---

### P2-03：书架空状态增强

**现状**：空状态显示一个简单的提示和"新建作品"按钮，缺乏吸引力。

**产品定义**：
- 替换为有视觉吸引力的空状态页
- 显示 3 个示例项目卡片（"一键创建示例项目"）
- "从模板开始"卡片引导

**处理方式**：做

---

### P2-04：底部状态栏（字数统计持续显示）

**现状**：字数在文档编辑器底部仅作为文本显示，在切换对象时不更新，且在其他标签页中不可见。

**产品定义**：
- 在底部添加全局状态栏（类似 VS Code）：
  - 左侧：保存状态指示器（P1-05）
  - 右侧：当前对象字数 / 项目总字数
  - 正典统计：已收录/未收录对象计数
- 状态栏在所有导航标签页中保持一致

**处理方式**：做

---

### P2-05：WikiLink 创建后自动导航

**现状**：在文档中双击不存在的 `[[对象名]]` 弹出创建气泡，创建后用户自动留在当前文档，新创建的对象被选中但没有导航到它。

**产品定义**：
- 在创建气泡中添加复选框："创建后打开"
- 默认勾选，用户可取消
- 创建后自动跳转到新对象的文档视图，同时在底部显示"已返回到之前位置"的返回按钮

**验收标准**：
- 创建对象后自动跳转到新对象文档
- 用户可以通过返回按钮回到之前的上下文

**处理方式**：做

---

### P2-06：框选（Lasso Select）

**现状**：多选仅通过 Ctrl+Click 一个一个点击。在需要批量移动或操作时效率低。

**产品定义**：
- 在选择模式下，点击空白处拖拽出现矩形选框
- 选框内节点全部选中
- 选中后可以批量拖拽移动
- 配合 Ctrl 可以追加选择

**验收标准**：
- 拖拽选框覆盖的节点全部高亮选中
- 选中后可批量拖拽

**处理方式**：做

---

### P2-07：自动布局锁定

**现状**：`autoLayoutRun` 和 `gridPlacedRun` 使用 ref 确保只运行一次，但用户手动调整位置后，如果触发 `handleAutoLayout` 或 `activeTab` 切换，用户的布局被覆盖。

**产品定义**：
- 在画板上方添加"锁定布局"开关（🔄 图标按钮）
- 锁定后自动布局按钮变灰，双击创建新对象时使用当前可用空白位置而非重新布局
- 解锁后可以触发自动布局

**验收标准**：
- 锁定状态下自动布局不可触发
- 锁定状态下用户拖拽的位置不会被任何自动操作覆盖

**处理方式**：做

---

### P2-08：设定推演图分区可拖拽

**现状**：设定推演图将对象按状态分区显示在固定区域，对象以列表形式展示（不是可拖拽节点）。用户无法手动调整对象在分区内的位置，也无法将对象拖拽到其他分区。

**产品定义**：
- 将设定推演图的分区中的列表项改为可拖拽节点
- 拖拽节点跨分区边界时自动更新对象的状态（如从"问题区"拖到"已锁定区" → 状态变为"锁定"）
- 分区内节点可以任意排列位置

**验收标准**：
- 设定推演图中的节点可以拖拽移动
- 跨分区拖拽更新对象状态

**处理方式**：做

---

### P2-09：时间线拖拽重排

**现状**：时间线的事件顺序在 `TIMELINE_EVENTS` 中硬编码，用户无法拖拽调整事件顺序或从待排事件区拖拽事件到时间线上。

**产品定义**：
- 时间线上的事件节点支持水平拖拽重排
- 待排事件区的事件可拖拽到时间线上指定位置
- 拖拽后更新 `narrative_units` 的 `order` 字段
- 时间轴自适应事件数量

**验收标准**：
- 用户可以通过拖拽任意调整时间线事件顺序
- 从待排区拖拽到时间线的事件自动获得位置

**处理方式**：做

---

### P2-10：设定集收录引导

**现状**：设定集页面顶部有一个简短的说明"设定集仅显示正典等级 > 未收录 的对象"，但用户如果发现设定集是空的，不知道如何"收录"对象。

**产品定义**：
- 设定集空状态时显示显式引导："还没有已收录的设定。在文档视图中选择一个对象，在右侧面板点击『收录为设定』"（附截图或图标指引）
- 在对象列表中添加"收录状态"列或图标
- 在设定集顶部的筛选器下方添加"如何收录？"帮助链接

**验收标准**：
- 设定集为空时，用户知道去何处执行收录操作
- 用户可在设定集中看到所有已收录对象的状态

**处理方式**：做

## P3 — 暂缓（未来版本）

### P3-01：作品封面自定义

允许用户为书架上的作品卡设置自定义封面（上传图片或选择渐变色）。**暂缓**原因：v1.2 聚焦核心体验，封面属于个性化增强。

### P3-02：模板自定义

允许用户修改对象模板的默认内容、标签和状态。**暂缓**原因：当前硬编码的 8 个模板覆盖了基本需求，模板编辑器需要独立的 UI 设计。

### P3-03：模板库选择

从模板库中选择预设的世界设定（如"赛博朋克城市"、"中世纪王国"）作为新项目的起点。**暂缓**原因：功能依赖模板编辑器（P3-02）和内容运营，v1.2 不做。

### P3-04：图片插入/附件

允许在文档中插入图片和附件文件。**暂缓**原因：需要文件管理 UI、存储方案和 Tauri file system API 集成，工作量大。

### P3-05：连接线编辑增强

当前连接线在 CanvasView 中用 SVG 绘制，编辑体验有限。未来可考虑增加折点编辑、曲线样式选择、连接线标签双击编辑等功能。**暂缓**原因：基础连线功能已可用。

### P3-06：正典手册/系统文档

内建帮助系统文档，解释正典系统、画板操作、WikiLink 等核心概念。**暂缓**原因：v1.2 用引导层替代，完整文档系统留给专门的"内建帮助"版本。

### P3-07：快捷键列表/帮助

展示和自定义快捷键。**暂缓**原因：核心功能快捷键已有（如 Ctrl+K 搜索规划在 P1-08），完整的快捷键查看器留给帮助系统。

## 各功能产品定义

### 功能组 A：存储与数据安全（P0）

**核心思路**：从"fire-and-forget"模型迁移到"本地优先 + 可靠同步"模型。

数据流：
```
用户操作 → 本地状态更新（乐观） → 同步队列（IndexedDB持久化） → 后台同步（指数退避重试）
                                                                            ↓
                                                                        成功 → 从队列移除
                                                                        失败 → 3次后标记为"待手动处理"
```

架构变更：
- 在 `App.tsx` 中引入 `SyncManager` 概念（类或 hook）
- 所有 `api.*` 调用通过 `SyncManager` 路由
- 引入 `App.tsx` 级别的 `queue` 状态，可在 UI 中显示待同步数量

不做的：
- 实时协作（多人编辑）
- 云端备份

### 功能组 B：编辑与写作体验（P1-03, P1-04, P1-05, P2-04, P2-05）

**核心思路**：让写作体验达到专业工具标准。

- Markdown 作为规范存储格式（P1-03）
- 单一工具栏 + 底部状态栏（P1-04, P2-04）
- 保存状态可感知（P1-05）
- WikiLink 创建流程闭环（P2-05）

### 功能组 C：画板交互增强（P1-06, P2-06, P2-07, P2-08, P2-09）

**核心思路**：让可视化工具真正做到"可视化可交互"。

- 缩放控件（P1-06）
- 框选（P2-06）
- 布局锁定（P2-07）
- 分区拖拽（P2-08）
- 时间线拖拽（P2-09）

### 功能组 D：新用户引导（P1-01, P1-02, P1-07, P2-03, P2-10）

**核心思路**：让新用户在 5 分钟内完成"创建 - 写作 - 链接 - 收录"的完整闭环。

- 创作向导（P1-01）
- 首次启动引导（P1-02）
- 正典系统引导（P1-07）
- 设定集收录引导（P2-10）
- 空状态增强（P2-03）

### 功能组 E：导航与发现（P1-08, P2-01, P2-02）

**核心思路**：让用户在项目中快速找到需要的信息。

- 全局搜索（P1-08）
- 书架管理增强（P2-01, P2-02）

## Out of Scope

1. **多人协作与实时同步** — v1.2 是单机工具。不引入任何多人编辑或 WebSocket 同步。
2. **云端账户与登录系统** — 没有用户系统。没有云端备份（本地导出替代）。
3. **导出为出版格式** — 不包含 EPUB/PDF 导出。只支持 JSON/Markdown 原始数据导出。
4. **AI 辅助写作** — 不集成 LLM 接口来生成内容。织梦机是组织工具，不是生成工具。
5. **模板市场** — 不搭建用户上传/分享模板的平台。
6. **移动端适配** — 不优化移动端体验。Tauri 桌面应用。
7. **插件/扩展系统** — 不提供第三方扩展 API。
8. **地图/地理系统** — 不提供地图绘制或地理坐标系统（世界地图功能）。
9. **关联阅读/推荐系统** — 不基于用户内容做推荐。
10. **性能重构** — 不进行全面的 Rust 后端重构或 WebWorker 迁移。性能问题在现有架构内解决。
11. **国际化/多语言支持** — v1.2 聚焦中文用户。不引入 i18n 框架。
12. **数据统计仪表盘** — 不提供写作统计、速度追踪、目标管理等元功能。
13. **批量操作框架** — 暂不实现全系统的批量选择/编辑/删除。只在画板支持多选拖拽。
14. **正典评审工作流** — 不在应用内实现多人评审/批准流程。正典管理仍是单人操作。

## 竞品与差异化

### 竞品参考

| 竞品 | 优势 | 不足 | 织梦机差异化机会 |
|------|------|------|----------------|
| **World Anvil** | 成熟的 wiki 式世界构建、社区生态 | Web 依赖、加载慢、付费墙后功能少 | 本地原生性能、离线可用、无订阅亦可全功能 |
| **Campfire Pro** | 功能全面、有叙事模块 | UI 老旧、仅英文、价格高（$75） | 现代 UI、中文优先、更合理的定价 |
| **Obsidian + 插件** | 灵活、Markdown 原生、免费 | 非专门世界构建工具、需自行配置插件 | 开箱即用的世界构建体验、Canvas 三视图专业化 |
| **LegendKeeper** | 美观的交互地图、沉浸式阅读 | 重度依赖 Web、协作偏重 | 写作者的私人工作台而非展示平台 |
| **Scrivener** | 专业的写作工具 | 世界构建功能薄弱、无可视化关联 | 画板 + WikiLink 填补了写作工具的世界构建空白 |

### 织梦机的核心差异化

1. **本地优先** — Tauri 原生应用，数据存储在本地 SQLite。不依赖网络，没有订阅锁定的数据。
2. **三视图画板** — 角色关系图（力导向布局 + 自由连线）、时间线（横向时间轴 + 拖拽排期）、设定推演图（状态分区 + 五区流转）。这不是通用的流程图工具，而是为世界构建量身定制的可视化范式。
3. **正典系统** — 4 级正典等级（未收录/草案正典/项目正典/核心正典）加上 6 种状态（占位/草稿/待定/待验证/锁定/废弃）。这不是简单的标签，而是一个结构化的确定性管理框架，让创作者对自己的世界设定有清晰的可信度认知。
4. **判断记录** — 每次锁定/废弃/提升正典都附带原因和时间戳。这是世界构建的审计日志，解决"为什么我当初做了这个设定"的遗忘问题。
5. **WikiLink 系统** — 植入 TipTap 编辑器的双向链接系统，双击 `[[对象名]]` 即可创建引用或创建新对象的闭环。引用和被引用关系在 Inspector 中可见。
6. **中文原生** — 整个 UI 和概念体系从中文学创作习惯出发（体裁选择、CJK 字数统计、拼音搜索），而非英文工具的翻译版。

### 不做差异化的点

- 不做 AI 生成（已有太多 AI 写作工具，织梦机专注"组织和思考"而非"生成"）
- 不做社交/社区（已有 World Anvil 和 Kanka 覆盖）
- 不做出版格式（已有 Scrivener 和 Vellum 覆盖）
- 不做多人协作（v1.2 阶段验证单人价值）
