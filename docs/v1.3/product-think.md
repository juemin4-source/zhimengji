# 织梦机 v1.3 — AI 对话与写作体验结合的产品思考

> 编制：product-manager
> 日期：2026-07-08
> 版本：v1（产品思考，非最终 spec）

---

## 一、核心洞察

**对话不是 AI 的边界，编辑才是。** v1.3 必须先回答"AI 生成的内容如何在对话流中被用户直接编辑和收录"，而非先把对话界面做得像 ChatGPT。如果用户必须离开对话去另一个页面编辑，AI 就从写作助手退化成了聊天玩具。

**AI 的触发点不在 AI 页面里，而在写作的每个自然停顿处。** v1.3 的全页对话界面是必要的起点，但它的终极形态是逐渐溶解到编辑器中的——用户不应该"去找 AI"，AI 应该在用户需要的时候出现。

---

## 二、五个维度的分析

### 2.1 对话即编辑（Conversation is Editing）

#### 现状

当前原型（ai-assistant.html）中，文档卡片是只读的。底部的 "编辑" 按钮仅弹出 alert。用户看到 AI 生成的设定后，无法直接在对话中修改。

#### 分析

这是 v1.3 最关键的设计决策。AI 对话在写作工具中的独特价值不是聊天，而是**生成→编辑→收录**的闭环效率。如果这个环断了，AI 对话就成了一个独立的"外挂"，和织梦机的核心写作体验脱节。

**文档卡片在对话中的形态应该分层：**

| 层级 | 形态 | 触发条件 | 交互 |
|------|------|---------|------|
| 预览 | 精简卡片（现有形态） | AI 首次生成 | 可读、可展开 |
| 内联编辑 | 卡片内字段直接编辑 | 用户点击编辑按钮或双击卡片内容 | 就地修改，即时保存 |
| 全屏编辑 | TipTap 编辑器弹出 | 用户点击"在编辑器中打开" | 完整的写作环境 |

**核心设计原则：**

1. **编辑不跳转** — 用户可以在对话流中完成 80% 的编辑操作（修改标题、调整描述、增减字段）。只有当需要大幅度改写时才弹出全屏编辑器。
2. **修改即同步** — 用户在卡片中的每次修改，自动通过同步队列写入项目数据。AI 在后续对话中能感知到用户的修改（"我注意到你把天眼的背景从 2078 年改到了 2081 年，是否要同步更新相关的人物设定？"）。
3. **修改历史可追溯** — 卡片底部显示"已编辑"标记和最后编辑时间，点击可查看修改记录。

**AI 感知用户修改的机制：**

```
用户修改卡片内容
  → 触发 onUpdate 事件
  → 更新 world_objects 表
  → 同时记录 diff 到 conversation context
  → 下一条 AI 请求时，在 system prompt 注入：
    "用户手动修改了以下内容：{diff}。请基于修改后的状态继续。"
```

#### 关键风险

- 编辑链路过长：如果内联编辑太复杂，用户还是想跳转页面
- 同步冲突：用户在对话中编辑的同时 AI 正在生成新版本
- 解决：内联编辑只改文本字段（名称、描述），不处理结构变更（删除字段、重新排序），结构变更走全屏编辑器

### 2.2 大纲和对话的联动

#### 现状

原型中已实现基础的"点击大纲对象 → 对话中对应文档卡片高亮"（focus system）。但这是单向的：大纲 → AI。

#### 分析

大纲树和 AI 对话之间的联动应该是一条**双向桥**，而不是单向公路。目前原型已经有了桥的一半，另一半（AI 创建对象 → 大纲实时更新、拖拽进对话）还未实现。

**三层的联动深度：**

```
第一层（v1.3 必须）：点击聚焦
  点击大纲对象 → AI 切换上下文焦点
  AI 创建对象 → 大纲自动添加节点

第二层（v1.3 建议）：状态同步
  对话中编辑卡片 → 大纲中的标题/状态实时更新
  大纲中修改对象 → 对话中的卡片同步刷新

第三层（v1.4+）：拖拽交互
  从大纲拖拽对象到对话 → AI 理解"我在讨论这个"
  从对话拖拽卡片到大纲 → 建立层级关系
```

**"聚焦"的语义需要扩展：**

当前原型的 focus 只是高亮卡片 + 滚动到位置。但用户点击大纲对象的真实意图是："让 AI 理解我现在要讨论这个东西"。所以 focus 应该触发：

1. 在对话中插入一条系统消息（对用户不可见）：`[Context Focus] 用户正在关注对象"天眼纪元"。后续对话以此对象为主要上下文。`
2. 将对象的完整数据加载到 AI 的 context 中
3. 如果之前 focus 了另一个对象，将旧对象的 context 降级为"background"，新对象升级为"active"

**大纲实时更新的架构方案：**

```
AI 生成文档卡片
  → 卡片渲染时调用 createWorldObject (写入项目数据)
  → 大纲组件监听 project data change event
  → 大纲树自动添加新节点
  → 如果是新创建的角色，自动展开对应文件夹
  → 节点标题闪烁 2 秒作为视觉反馈
```

### 2.3 写作流中的 AI 触发点

#### 分析

这是织梦机区别于"ChatGPT 套壳"的核心所在。v2.0 北极星说的"越用越懂你"不是通过对话窗口实现的，而是通过 AI 在写作的自然时刻出现实现的。

**触发点矩阵：**

| 触发时机 | 触发条件 | AI 行为 | v1.3/v1.4 |
|----------|---------|---------|-----------|
| 光标暂停 | 光标在编辑器中停留 > 5 秒，且当前段落不足 50 字 | 右下角出现轻量建议："需要我帮你展开吗？" | v1.4 |
| 创建新对象 | 用户通过 [[WikiLink]] 创建新对象 | 自动弹出"是否使用设定模板"选择器 | v1.4 |
| 段落完成 | 用户在编辑器中写完一个段落（按下两次 Enter） | 不打扰，但在侧边栏显示一致性检查计数 | v1.4 |
| 选中文本 | 用户选中编辑器中的一段文本 | 浮动工具栏出现 AI 按钮：改写/扩写/缩写 | v1.4 |
| 文中 `@AI` | 用户在编辑器中输入 `@AI` | 光标处内联弹出 AI 输入框（不离开编辑器） | v1.5 |
| 打开 AI 页面 | 用户主动导航到 AI 助手页面 | 全页对话模式，完整文档卡片嵌入 | v1.3 |

**关键判断：**

v1.3 不能做上述所有触发点。这会分散焦点且风险太高。但 v1.3 的架构必须为这些触发点做好准备：

1. `@AI` 内联触发要求编辑器（TipTap）支持自定义插件扩展——v1.3 需要验证 TipTap 插件架构是否支持
2. 段落完成检测要求编辑器有 onUpdate 回调的防抖机制
3. 选中文本的浮动工具栏要求编辑器 API 能获取选区位置
4. 光标暂停检测需要 idle detection + cursor position tracking

**建议的架构准备：**

```typescript
// src/hooks/useWritingTriggers.ts — v1.3 预埋，v1.4 启用
interface WritingTrigger {
  type: 'cursorPause' | 'newObject' | 'paragraphComplete' | 'textSelection' | 'inlineAI';
  context: {
    position?: EditorPosition;
    selectedText?: string;
    surroundingText?: string;
    objectType?: string;
  };
  handler: (context: WritingTrigger['context']) => void;
}
```

**v1.3 唯一需要做的触发点**是当用户在 AI 对话中生成了文档卡片后，提供"插入到编辑器"的按钮——这个按钮在当前原型中已经存在（"收录为设定"），但它应该更明确地导向编辑器中的当前位置。

### 2.4 上下文感知

#### 分析

这是 AI 对话体验"聪明不聪明"的分水岭。如果用户每次都要解释"我的故事是关于什么的"，AI 对话就和 ChatGPT 没区别。织梦机的 AI 应该自动知道当前项目的全貌。

**上下文层级架构：**

```
Token 预算: ~6000 tokens (byok_call_llm 默认)

Level 1 — Active Context（~2000 tokens，始终加载）
  ├ 当前 focus 对象的完整数据
  ├ 当前 focus 对象的 relations（谁关联谁）
  ├ 最近 5 条对话历史
  └ 项目名称/类型/基调

Level 2 — Project Context（~2000 tokens，按需加载）
  ├ 正典等级 ≥ 项目正典 的所有对象（标题 + 摘要）
  ├ 最近的编辑历史（最后修改的 10 个对象）
  ├ 风格画像摘要（style_profile 的 condensed 版本）
  └ 全局设定规则（如"这是一部科幻小说，时间线在 2100 年"）

Level 3 — Library（~2000 tokens，懒加载）
  ├ 正典等级 = 草案正典 的对象（仅标题）
  ├ 所有对象的引用关系图摘要
  └ 章节列表 + 状态

Token 紧张时降级策略：
  1. 先降 Level 3 到 500 tokens（只保留标题列表）
  2. 再降 Level 1 的 relations（只保留直接关联）
  3. 不降级：项目名称、当前 focus 对象的基础信息
```

**Focus 系统的上下文含义：**

原型的 focus 系统目前只是 UI 高亮。它应该扩展为一个**上下文管理系统**：

```typescript
interface FocusContext {
  // 当前聚焦的对象
  activeObject: { id: string; type: string; data: WorldObject };
  
  // 相关对象（activeObject 引用或被引用的对象）
  relatedObjects: WorldObject[];
  
  // 历史焦点（用于"回到之前的话题"）
  focusHistory: string[];  // object IDs
  
  // 上下文加载状态
  loadingLevel: 'minimal' | 'standard' | 'full';
  tokenEstimate: number;
}
```

当用户点击大纲中的"天眼纪元"时：
1. 加载天眼纪元的完整数据 → Level 1
2. 加载关联对象（破晓组织、天眼中枢） → Level 2
3. 加载其他高正典设定 → Level 3
4. 在 AI 的 system prompt 注入当前 focus 信息

**AI 对话需要显示当前的上下文状态：**

原型底部的 token 用量条应该扩展为上下文指示器：

```
[tokens: 6.2K / 15K] [聚焦: 天眼纪元] [+3 关联对象] [124/178 设定可用]
```

让用户知道 AI 知道什么，不知道什么。这在用户感觉"AI 好像忘了我的设定"时特别重要。

### 2.5 v1.3 边界

#### 核心判断

v1.3 的核心是：**AI 对话能读写本地数据**。它的成功标准不是"AI 多聪明"，而是"AI 生成的内容能不能无缝进入用户的项目中"。

从原型路线图看，v1.3 原定是"多模型切换 + 用量监控"。但现有的原型（ai-assistant.html）已经远远超过了这个范围——它有了全页对话、文档卡片生成、大纲树联动。这说明团队已经在实践中把 v1.3 的范围扩大了。

我的建议是：**承认范围扩大，但设一个清晰的"拉闸点"**。

#### 范围建议

**Must Have（v1.3 上线必须）**

| 功能 | 说明 | 依赖 |
|------|------|------|
| 全页 AI 对话 | ChatGPT 风格的对话界面，会话管理（新建/切换/历史） | 已有原型 |
| 文档卡片嵌入 | AI 生成的内容以文档卡片形式嵌入对话流 | 已有原型 |
| 卡片内联编辑 | 用户可以在对话中直接编辑卡片内容（标题、描述、section） | BYOK 写入 + 同步队列 |
| 大纲-对话单向联动 | 点击大纲对象 → 对话 focus 该对象（已有）AI 创建对象 → 大纲更新（新增） | 项目数据监听 |
| 基础上下文加载 | 对话自动加载当前项目的名称、类型、高正典设定（Level 1+2） | BYOK 层 context injection |
| 多模型切换 | 用户切换底层模型（GPT/Claude/本地） | BYOK 层 model_router |
| Key 管理 | 加密存储 API Key、连接测试、多 Key 切换 | BYOK 层 key_manager |
| 用量监控 | 实时 token 计数、用量进度条 | BYOK 层 usage_tracker |
| 收录到项目 | 文档卡片一键收录为项目对象（创建 world_object 记录） | 收录流程 |

**Should Have（v1.3 排期中，可移交 v1.4 但不难）**

| 功能 | 说明 |
|------|------|
| 大纲-对话双向联动 | 对话中编辑卡片 → 大纲实时更新标题/状态 |
| 上下文指示器 | 对话中显式显示 AI 当前知道什么对象 |
| 对话历史持久化 | 关闭页面后重新打开，对话记录不丢失，按项目分组 |
| 卡片全屏编辑 | 点击"在编辑器中打开"弹出 TipTap 全功能编辑器 |

**Deferred（v1.4+）**

| 功能 | 规划版本 | 说明 |
|------|---------|------|
| 编辑器内 AI 触发点（光标暂停、段落完成、选中文本） | v1.4+ | 依赖 WritingTrigger 架构，且需要稳定的 editor plugin API |
| `@AI` 内联对话 | v1.5 | 要求编辑器深度集成，v1.3 不做 |
| 拖拽大纲 → 对话 | v1.4 | 交互设计需要验证，体验风险较高 |
| 全量 smart context management | v1.4 | Level 3 懒加载 + token 预算算法需要实验 |
| Agent 框架（Architect/Writer/Reader/Inspector） | v1.4 | 独立版本，v1.3 只做基础对话 |
| 风格画像自动建立 | v1.4 | 依赖 style-profile-schema 实现 |
| 一致性检查 | v1.6 | 依赖 Inspector Agent |

**v1.3 的"不做什么"**

1. 不做 Agent 编排（那是 v1.4 的核心）
2. 不做编辑器内嵌 AI（那是 v1.5 的核心）
3. 不做风格画像自动学习（那是 v1.4 的核心）
4. 不做 skill 系统（那是 v1.4 的核心）
5. 不做画板 AI 化（那是 v1.6 的核心）
6. 不做全量项目上下文自动加载（v1.3 只做基础 Level 1+2，Level 3 等 token 预算算法成熟后做）

## 三、对原型的修改建议

### 3.1 文档卡片增加内联编辑能力

当前原型的文档卡片是纯展示。建议改为：

- 点击"编辑"按钮 → 卡片字段变为可编辑输入框
- 标题区：text input
- Body 区：textarea（支持 Markdown 快捷语法）
- Section 区：每个 section 独立可编辑，支持增加/删除 section
- 底部新增"保存"和"取消"按钮，保存后调用 `updateWorldObject`
- 编辑状态下禁用"继续展开"按钮（防止冲突）
- 卡片左上角显示编辑状态：`已编辑` 标记 + 时间戳

### 3.2 大纲树的实时更新

当前大纲树是静态 HTML。建议：

- 将大纲树数据接入项目的 `world_objects` 数据源
- 当 AI 创建新对象时：
  1. 调用 `api.createWorldObject` 写入数据库
  2. 大纲树监听变更事件，自动新增节点
  3. 新节点闪烁 2 秒
  4. 如果新对象的类型匹配已有的文件夹（如"组织"），自动归入对应文件夹
- 当用户在对话中编辑卡片时：
  1. 调用 `api.updateWorldObject` 更新数据库
  2. 大纲树同步更新节点标题

### 3.3 上下文可视化和控制

当前原型的 token 用量条只是一个数字。建议：

- 在用量条旁边显示"已加载上下文"的概览
- 点击可以展开上下文详情面板，显示当前 AI 知道哪些对象
- 用户可以从上下文面板中移除某个对象（释放 token 给其他内容）
- 提供"手动添加上下文"入口：用户选择要传入 AI 的特定对象

### 3.4 "发送到编辑器"流程增强

当前原型有"收录为设定"按钮。建议增加：

- "发送到编辑器"按钮：将卡片内容插入到当前编辑器的光标位置
- 如果编辑器没有打开或不在文档页面，先保存为对象再提示用户跳转
- 卡片底部显示"已在编辑器"标记，避免重复插入

### 3.5 对话历史管理

当前原型只有"新对话"按钮，没有历史对话管理。v1.3 应该为后续的对话历史系统打好基础：

- 每次对话分配一个 sessionId
- 对话按项目分组（项目 A 的对话不会混入项目 B 的）
- 每个对话生成一个自动标题（AI 根据第一条消息生成）
- 对话列表在 AI 页面的左侧或侧边栏底部

## 四、开放问题

### Q1：卡片编辑的"保存"到底做什么？

当用户在对话中编辑文档卡片并点击"保存"时，这个操作应该：
- A) 更新对话中的卡片内容（本地状态） + 异步写入数据库
- B) 先写入数据库，再从数据库读取最新版本刷新卡片
- C) 写入对话历史 + 数据库双向

**我的倾向：** A。对话中的卡片是"可编辑的预览"，不是数据库的直接镜像。第一次保存是本地更新 + 数据库写入，后续 AI 回复时从数据库读取最新版本。这样可以避免在对话中频繁读取数据库造成的延迟。

### Q2：内联编辑的深度边界在哪？

用户应该能在对话中编辑到什么程度？
- 编辑标题和描述文本 ✓
- 新增/删除 section ✓
- 修改 section 内容 ✓
- 删除整张卡片 ❌ （删除走对象管理，不在对话中操作）
- 修改卡片类型（world → org） ❌ （类型是结构属性，走对象编辑页面）
- 重新排序卡片内的 sections ❌ （v1.3 不做，留给全屏编辑器）

### Q3：当 AI 正在生成时用户编辑了卡片，如何处理？

场景：AI 正在生成"天眼纪元展开版"（耗时 2 秒），同时用户编辑了"天眼纪元基础版"的标题。两个操作同时写入数据库，可能产生冲突。

方案：
- 乐观更新：用户编辑先写入本地状态并标记为"待同步"
- AI 生成完成后，检查用户是否有未同步的编辑
- 如果有，在 AI 的 response 中加入备注："我注意到你在生成过程中修改了设定，我的回复基于原始版本。是否要我基于修改后的版本重新生成？"

### Q4：大纲 click → focus 到对话 vs focus 到编辑器？

场景：用户正在编辑器中写作，点击大纲中的对象。应该：
- A) 跳转到 AI 对话页面并 focus 该对象
- B) 在编辑器侧边栏显示该对象的摘要

**我的倾向：** 区分来源路径。如果用户从 AI 页面点击大纲，保持在 AI 页面并 focus。如果用户从编辑器页面点击大纲，在编辑器侧边栏显示摘要（不跳转页面）。未来（v1.5）当 AI 深入编辑器后，这个区分会自然消失。

### Q5：是否在 v1.3 引入"会话"的概念？

当前原型只有一个对话流。但用户可能想要：
- 多个并行的对话（一个讨论世界观，一个讨论角色）
- 每个对话有独立的上下文 focus

**我的倾向：** v1.3 引入基础的会话管理（新建/切换/历史列表），但不做并行对话。并行对话的上下文管理复杂度太高，留给 v1.4。

### Q6：多模型切换对对话一致性有什么影响？

用户先用了 Claude 3.5 Sonnet 创建了"天眼纪元"，然后切换到 GPT-4o 继续讨论。GPT-4o 是否能看到 Claude 3.5 生成的内容？

**原则：** 对话历史的可见性不依赖模型。切换模型只是改变"下一个回复"的生成引擎，不影响对话的上下文。模型切换的 UI 应该强调这一点："切换模型不影响之前的对话内容"。

## 五、架构影响总结

v1.3 的 AI 对话功能对现有织梦机架构的影响：

| 现有模块 | 影响 | 变更内容 |
|---------|------|---------|
| world_objects 表 | 读取 | AI 生成新对象时写入，编辑时更新 |
| api.createWorldObject | 已存在 | 卡片保存时调用 |
| api.updateWorldObject | 已存在 | 卡片编辑时调用 |
| BYOK 层 (Rust) | 新增 | key_manager, model_router, usage_tracker |
| 同步队列 | 使用 | 卡片编辑的写入走同步队列（复用 v1.2 P0-01） |
| 对话历史存储 | 新增 | 新表 conversation_sessions + conversation_messages |
| 上下文注入 | 新增 | BYOK 层 inject_style_into_prompt 扩展为 inject_context |
| TipTap 编辑器 | 无影响 | v1.3 AI 对话不侵入编辑器（v1.4+ 才做） |
| 大纲树组件 | 更新 | 新增数据监听 + 实时更新能力 |
| 画板/设定集/判断 | 无影响 | v1.3 不改这些页面 |

**新增数据结构（对话历史）：**

```sql
CREATE TABLE IF NOT EXISTS conversation_sessions (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  title TEXT NOT NULL DEFAULT '',     -- AI 自动生成的对话标题
  model_id TEXT NOT NULL,             -- 创建此对话时使用的模型
  message_count INTEGER NOT NULL DEFAULT 0,
  token_count INTEGER NOT NULL DEFAULT 0,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

CREATE TABLE IF NOT EXISTS conversation_messages (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  role TEXT NOT NULL CHECK(role IN ('user', 'assistant', 'system')),
  content TEXT NOT NULL,
  -- 如果这条消息包含文档卡片关联，记录卡片对应的对象 ID
  related_object_id TEXT,
  tokens INTEGER NOT NULL DEFAULT 0,
  created_at INTEGER NOT NULL,
  FOREIGN KEY (session_id) REFERENCES conversation_sessions(id) ON DELETE CASCADE
);
```

## 六、退出标准

```
Gate: 对话即编辑
  [PASS] 用户可在对话流中编辑 AI 生成的文档卡片
  [PASS] 编辑后的内容同步写入项目数据库
  [PASS] AI 在后续对话中能感知到用户的编辑

Gate: 大纲联动
  [PASS] 点击大纲对象 → AI 上下文 focus
  [PASS] AI 创建新对象 → 大纲实时更新
  [PASS] 对话中编辑卡片 → 大纲同步更新

Gate: Key 管理和多模型
  [PASS] API Key 可加密存储、测试连接、删除
  [PASS] 模型切换不丢失对话上下文
  [PASS] 用量监控实时更新

Gate: 上下文感知
  [PASS] AI 对话自动加载项目基本上下文（名称/类型/正典设定）
  [PASS] Focus 系统明确告知 AI 当前在讨论什么
  [PASS] token 用量可视化

Gate: 体验
  [PASS] 用户从"新建对话"到"生成并收录一个设定"不超过 5 步
  [PASS] 对话历史按项目分组可回溯
  [PASS] 无 P0 缺陷
```
