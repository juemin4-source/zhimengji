## Four Review Dimensions

### 1. Scope Drift（范围漂移审查）

**总体判断：v0.3 的范围调整合理、有明确动机、且忠于产品定位。**

#### 1.1 版本新增与调整的合理性

| 变更 | 漂移评估 | 理由 |
|------|---------|------|
| 新增 v2.0-H RC Blocker Fix | ✅ **合理且必要** | D 轮缺口（AI三态路由、E2E、报告状态）是真实的主流程阻塞项。v2.0.1 的可用性测试需要一条真正可用的管线，不补缺口则探针失效 |
| 新增 v2.0.1 Usability Probe | ✅ **结构性问题修正** | 直接响应评估报告的 #1 结构风险——"4 个月方法论补完期间无用户反馈"。在方法论骨架建设前插入可用性探针，是正确顺序 |
| v2.1.0 Spike→MVP 升级 | ⚠️ **方向正确但范围扩大** | 从纯技术 Spike（1-2 周，原型标准）升级为全产品版本（4-5 周，交付标准），范围扩大约 3 倍。这是必要的——纯 Spike 的产出无法支持 v2.1.2 的价值验证。但需在后续审查中确认工程可行性 |
| v2.1.2 重新定义为 Value Probe | ✅ **核心改进** | 从"集成层"到"价值验证"是 PRD 质的飞跃。明确界定两探针的区别（见第 21 节），使产品路线有了明确的 Go/No-Go 决策节点 |
| v2.1.1 重新定义为 Method Integration | ✅ **合理** | 从"方法论补完"（无明确边界）到"方法论影响下游管线"（有明确边界），版本目标更加聚焦 |

### 2. Intent Mismatch（意图错配审查）

#### 2.1 季节对齐

v0.2 的一个核心问题是"春生不做春生该做的事"——在方法论补完 3-4 个月期间无用户反馈。

v0.3 做了明确的季节修正：
- 显式声明当前为**春生早段**
- 核心逻辑从"先方法论完美再上线"调整为"先上线找 PMF"
- v2.0.1（了解用户能不能用）早于 v2.1.0（方法论骨架）
- v2.1.2（验证产品有没有价值）作为 Go/No-Go 决策节点

**季节判断与版本路线的对齐度：v0.3 >> v0.2。评分从 ⚠️ 提升至 ✅。**

#### 2.2 "方法论是壁垒，不是广告"原则贯彻

新增的产品原则第 6 条在以下位置被正确执行：

| 位置 | 执行方式 |
|------|---------|
| v2.0.1 入口设计 | "一键速写入口不得提及方法论术语" |
| v2.0.1 Out of Scope | 明确列出"方法论术语在入口处出现"为 ❌ |
| v2.1.0 方法论原则 | "方法论术语对用户不可见"——用户看到的是自然语言问题 |
| A 星定位 | "入口只说'打开就写'，不说'请先学习我们的方法论'" |
| B 星定价 | "方法论溢价不单独列出，内嵌于点数价格中" |

**原则已落地为可执行的范围约束，不是空洞的声明。**

#### 2.3 两探针的区分表达

第 21 节（附录）是对两探针区别最完整的文档表达。关键要点：

### 3. Internal Consistency（内部一致性审查）

#### 3.1 版本依赖链

```
v2.0 ─→ v2.0-H ─→ v2.0.1 ─→ v2.1.0 ─→ v2.1.1 ─→ v2.1.2 ─→ v2.2 → v2.3 → v2.4 → v2.5 → v2.6
```

依赖关系由原来的平行分支（v0.2: v2.1.0 ∥ v2.1.1）改为**严格的线性依赖链**。每条边都有明确理由：

| 依赖边 | 理由 | 合理性 |
|--------|------|--------|
| v2.0 → v2.0-H | 主流程基线 → 补缺口 | ✅ 强依赖 |
| v2.0-H → v2.0.1 | 主流程可用 → 测可用性 | ✅ 强依赖——不可用的管线无法做可用性测试 |
| v2.0.1 → v2.1.0 | 可用性数据 → 指导方法论设计 | ✅ 合理——但注意这不是强技术依赖 |
| v2.1.0 → v2.1.1 | 方法论骨架 → 方法论集成 | ✅ 强依赖——v2.1.1 需要 v2.1.0 的数据模型 |
| v2.1.1 → v2.1.2 | 方法论集成就绪 → 价值验证 | ✅ 强依赖——B 组需要完整方法论管线 |

**依赖链无断裂、无循环依赖。比 v0.2 的并行结构更清晰。**

#### 3.2 版本编号一致性

当前编号体系：
```
v2.0 → v2.0-H → v2.0.1 → v2.1.0 → v2.1.1 → v2.1.2 → v2.2 → v2.3 → v2.4 → v2.5 → v2.6
```

**v0.1 Gate 条件 C-1（Phase vs v2.x 映射）已解决。** v0.3 继承了 v0.2 对旧 Phase 体系的废弃声明和显式对照表，旧 Phase 编号不再作为版本语言。

## Issues Found

### P0 级问题（必须解决）

### P1 级问题（强烈建议在执行中解决）

#### P1-1: v2.0.1 "一键速写入口"与管线路径之间存在验证盲区

**描述：** v2.0.1 的"一键速写"入口（跳过所有画板直接出正文）与产品的核心差异化（方法论管线）之间存在张力。如果 80% 的用户只用"一键速写"而不触碰画板，Usability Probe 的行为数据将无法回答"管线可用性如何"。

**风险：** 可用性探针测的不是核心产品体验，而是一个快速出口功能。

**建议：** 在 v2.0.1 设计阶段，明确划分两种测试路径：
- 路径 A（基线）：一键速写入口 → 快速出正文
- 路径 B（管线）：标准管线路径 → 用户主动选择走画板
- 两个路径分别收集数据，避免数据混合后无法解读。

#### P1-2: v2.1.0 从 Spike 升级为 MVP 的范围增加未对应削减

**描述：** v0.2 的 v2.1.0（Spike）预估 1-2 周，容忍已知瑕疵。v0.3 的 v2.1.0（MVP）预估 4-5 周，要求产品级质量。这是 3 倍的范围增加，但与此同时没有任何其他版本的路径缩短或削减来对冲。

**风险：** 如果 v2.1.0 实际需要 6-7 周（考虑到 4 个画板的同时修改），将直接挤压 v2.1.1 的 2-3 周时间窗口，进而影响 v2.1.2 的排期。

**建议：**
1. 在 v2.0-H 执行完成后，基于实际速度重估 v2.1.0 时间
2. 将画板③的 P1 项（天/地/人展开）明确标记为"可推迟到 v2.1.1"
3. 如果 v2.1.0 工时超出 5 周，考虑将画板④三档细度模式的精编模式推迟到 v2.1.1

#### P1-3: v2.0-H 的 E2E 验收是一条脆弱的单点依赖

**描述：** 整个链条的后续版本（从 v2.0.1 到 v2.6）都依赖 v2.0-H 的 `npm run accept:e2e` 通过。但该 E2E 基础设施"存在但未跑通"（文档原文），修复它本身就是 v2.0-H 的 P0 项之一。

**风险：** 如果 E2E 基础设施修复遇到根本性困难（如 Tauri driver 兼容性问题），v2.0-H 会卡住，而 v2.0-H 是后续所有版本的前置依赖——单点故障风险极高。

**建议：**
1. v2.0-H 启动时优先处理 E2E 基础设施（其他 4 项修复可以并行）
2. 如果 1 周内 E2E 不能通过，准备人工验收的备用方案
3. 为 E2E 基础设施修复设置独立的硬截止日期

#### P1-4: 两探针的"验证失败回路"未定义

**描述：** v2.0.1 定义了 6 个验收指标和收集方法，但未定义"如果指标全部不合格怎么办"。v2.1.2 定义了 Go/No-Go 标准，但 No-Go 后只说"退回到产品定义阶段"，未说明谁做、以什么流程做、花多久。

**风险：** 数据收集了但没有预定义的行动框架，容易导致"收集了 3 个月报告然后犹豫不决"。

**建议：** 在 v2.0.1 和 v2.1.2 的完成标准中增加验证失败时的默认动作：
- v2.0.1：如果 5 分钟达标率 < 50%，下一阶段优先修复而非继续方法论
- v2.1.2：如果 No-Go，设定 2 周分析期 + Chancellor 裁决时间点

### P2 级问题（建议关注）

#### P2-1: 设计规格文档的 MVP 范围未标注

v0.3 的文档索引（第 21.5 节）引用了完整的画板规格文档（brainstorm-spec、outline-spec、world-spec 等），但这些规格文档对应于完整方法论版本，而非 v2.1.0 MVP 版本所需的范围。执行 v2.1.0 时，团队需要知道"这些规格中的哪些功能属于 MVP、哪些属于后续版本"。

**建议：** 在 v2.1.0 开始执行前，在各规格文档中增加 MVP 范围标记（或创建一个 v2.1.0 scope matrix 做交叉引用）。

#### P2-2: v2.1.1 和 v2.1.2 的时间估计可能偏乐观

- v2.1.1（2-3 周）：5 项跨画板的深度数据集成工作，包括上游修改提示、前提→写作契约、结构节点→ChapterPacket 映射等。2 周似乎不足。
- v2.1.2（2 周）：A/B 测试基础设施 + 30 用户招募 + 实测 + 专家盲评 + 统计分析。在 2 周内完成实验设计到结果分析的全流程，对于第一次做这种测试的团队来说密度很高。

**建议：** 在实际执行中将预估上调 30-50%，并明确哪些项可以推迟。

## Notes for Chancellor

### 1. v0.3 是一份应对审查的高质量修订

v0.3 对 v0.1 Gate 的 4 个条件和评估报告的 6 个关键问题均做出了回应。这不是简单的文字修改——产品原则第 6 条、两探针分离、季节调整、丢失项裁决都是有实质内容的改动。文档质量从 v0.1 的 B+ 提升至 A- 级别。

### 2. 建议在 v2.0-H 完成后、v2.0.1 启动前进行一次中型评审

v2.0.1 的设计决策（一键速写 vs 管线路径的数据分离策略）将影响整个春生的数据收集质量。建议在此节点由 Scope Guardian 或 Product Manager 进行一次轻量审查（不需要 4 维度 Full Review）。

### 3. v2.1.0 是所有版本中风险最高的节点

尽管 v0.3 将 v2.1.2 也标为高风险，但 v2.1.0 的 3 倍范围扩大（Spike→MVP）使其成为实际执行风险最高的版本。如果 v2.1.0 出现延期，会连锁影响 v2.1.1 和 v2.1.2 的时间窗口，进而影响 v2.1.2 的 A/B 测试排期（涉及用户招募、专家可用性等不可控因素）。建议在 v2.0.1 完成后、v2.1.0 启动前，对 v2.1.0 做一次独立的可行性审查。

### 4. 文档中"方法论 Skills 注册"的依赖关系需要显式排期

方法论 Skills 注册（6 个方法论文件注册为 `.claude/skills/zhimengji/`）是 v2.1.0 启动的前置条件。当前文档中提到的"v2.1.0 启动前置条件：方法论 Skills 注册"出现在依赖关系图中但未指定具体时间。建议明确："方法论 Skills 注册必须在 v2.0.1 执行期间完成，不占用 v2.1.0 工时"。
