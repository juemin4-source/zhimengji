## 1. 产品北极星

### 1.1 一句话定义

**织梦机是一款 AI 管线化小说创作工具——用户和 AI 在同一块画板上工作，AI 走管线，用户在每站有否决权。**

它不是"AI 全自动写作工具"（Notion AI / Sudowrite），也不是"纯手工编辑器"（Scrivener / Ulysses）。它是两者之间的一条结构化管道：作者像导演，AI 是他的创作班子。AI 按画板顺序逐一输出结构化的创作素材，用户在每一站确认、修改或推翻。

### 1.2 最小闭环

> 一条前提句 → AI 走完 5 画板管线 → 用户在每个输出节点确认/修改 → 成文。

织梦机的最小可交付价值不是某个按钮或某个面板，而是这条管线的**端到端可运行性**。从模糊想法到成品正文，用户只需做"确认/修改"的轻量操作，重活（生成前提句、展开结构、填充设定、拆解细纲、撰写正文）由 AI 在画板约束下完成。

### 1.3 三大核心痛点

| 痛点 | AI 写作工具的现状 | 织梦机的解法 |
|------|------------------|-------------|
| **上下文丢失** | AI 对话窗口越长越乱，前文角色/设定在 10 轮对话后开始混淆 | 四层自包含细纲包：每章压缩为"写作契约 + 设定快照 + 压缩叙事 + 执行层"的完整包，任意 AI 拿到即可正确写作 |
| **幻觉/不可控** | AI 擅自创建角色、使用未设定规则、写出"角色知道不应该知道的信息" | 知识边界检测器 + 临时假设机制：AI 遇到设定空白只能标记为"临时假设"，不得静默补全 |
| **方法论缺失** | 竞品只提供"帮我写"一个按钮，没有从想法到成文的结构化方法论引导 | 五画板管线 + 12 个方法论技能（八字六变/十二时位/世界构件/人物屠龙术/七诊/八体），用户在整个创作过程中被方法论赋能而非替代 |

### 1.4 生态位定位

织梦机的生态位不能用"功能对比表"来描述——因为它本质上不是"又一个 AI 写作工具"。它的竞争对手不是 Novelcrafter 或 Dabble，而是"空白页面带来的创作焦虑"和"AI 管线的上下文丢失"。要理解这个生态位，需要看两个维度：

**维度一：方法论深度**——产品是否内置了一套可执行的创作方法论。
**维度二：AI 管线化程度**——产品是否将创作流程从想法到正文组织为结构化的 AI 驱动管道。

```
                          方法论深度（高）
                               │
                               │
               World Anvil     │     织梦机 ⭐
               （设定有方法论  │     （方法论 x 管线 = 无人区）
                但无 AI 管线） │    
                               │
  ────────────────┼─────────────────────────── AI 管线化程度
                               │
       Scrivener / Ulysses     │     Novelcrafter
       （纯手工，无 AI）       │     （有 AI 但无方法论）
                               │     Dabble
                               │     （轻 AI，无方法论）
                               │
                          方法论深度（低）
```

### 1.5 A/B 星定位与造局分析

#### A 星（入口/吸引端）

| 维度 | 定义 | 依据 |
|------|------|------|
| **主属性** | **金（刀子/工具）** | 用户来织梦机是为了完成"写小说"这个明确任务。入口承诺是工具效率——"输入想法，输出章节" |
| **辅属性** | **水（浪潮/AI）** | AI 管线是钩子——"AI 帮您写"是吸引尝鲜者的杠杆。但此属性的持续性存疑：当 AI 工具从新奇变为标配时，水属性会消退 |
| **核心原则** | **方法论是壁垒，不是广告** | 用户不应在入口处感知到方法论。方法论藏在管线背后。入口只说"打开就写"，不说"请先学习我们的方法论" |
| **A 星一句** | **"输入想法，输出章节——AI 在背后帮你构建创作世界"** |

#### B 星（出口/变现端）

| 维度 | 定义 | 依据 |
|------|------|------|
| **主属性** | **金（订阅/点数）** | SaaS 订阅式，点数消耗。用户为持续使用工具的价值付费 |
| **B 星一句** | **"写一章花 5 点，免费额度够你完成第一章——当你需要写完整本书，就该付费了"** |

#### 造局检查（25 局）

```
A 星: 金（工具效率） + 辅水（AI 浪潮）
B 星: 金（订阅变现）
─────
局类型: 同局（金金）
局名: 绝对零度局
判词: 绝对零度，极致效率。冷到极致，没有对手能跟上你的节奏。
```

| 维度 | 评估 |
|------|------|
| 能量流向 | **通顺**——用户为工具而来，为工具付费。没有转化损耗 |
| 脆弱性 | **单一化风险**——A 和 B 都是金，意味着产品只有"效率工具"一个维度。如果出现一个同样效率但更便宜的竞品，织梦机的基础不牢固 |
| 辅助水属性的作用 | AI 浪潮（水）只影响 A 星的短期获客，不改变 B 星的结构。水是季节性的——当 AI 浪潮消退，产品的 A 星减为纯金，竞争压力增大 |
| 方法论的角色 | 方法论是最终建立品牌溢价（火）的根基，但在当前阶段（春生），方法论是隐形壁垒，不是获客工具 |
| 破局建议 | 当前不破局（同局对早期产品是好局）。当产品进入夏长阶段后，应考虑在 B 星引入**火（品牌溢价）**——通过方法论壁垒建立溢价能力 |

**造局结论：同局（金金），干净但单一。建议当前聚焦执行，不做局调整。方法论作为长期壁垒储备，在入口处隐去。**

## 2. 产品原则更新

### 2.1 核心设计原则（v0.3 更新版）

| 原则 | 含义 | 影响 |
|------|------|------|
| **前期约束，后期自由** | 画板①→②→③严格串行不可跳过；画板④→⑤推荐串行但允许宽松 | 数据依赖链驱动 UI 导航：未就绪画板可点击查看说明，但不可编辑 |
| **追问驱动，非填空驱动** | 每个画板的交互核心是"问题"而非"字段" | 用户看到的是自然语言问题，不是字段名。例如："这个世界是什么颜色？"而非"美学辨识度：[输入]" |
| **AI 是填槽助手，不是创作主体** | AI 读取上一阶段输出，填当前阶段的"槽" | AI 的产出必须标记"AI 生成"，用户可以逐项确认/修改/拒绝 |
| **四层自包含** | 每章细纲打包为自包含包，不依赖 AI 上下文记忆 | 导出到任意 AI 工具（DeepSeek / ChatGPT）都能写出正确正文 |
| **可逆设计** | 正向：想法→正文；反向：正文→分析全部画板 | 反向数据标记 source 来源，所有分析结果可修改 |
| **方法论是壁垒，不是广告**（新增 v0.3） | 方法论是织梦机的核心竞争力，但方法论术语在入口处不可见 | v2.0.1 入口只承诺"快速写作"，不提及方法论。方法论在用户自然进入画板后才逐步揭示。AI 生成时方法论作为底层管线运行，用户感知到的是"结果变好了"，而非"你正在使用方法论" |

### 2.2 获客假设（v0.3 新增）

以下假设在春生阶段待验证，不应作为确定性的产品策略：

| 假设 | 验证方法 | 验证版本 | 优先级 |
|------|---------|---------|--------|
| 用户因为"AI 输出更符合我的设定"而选择织梦机而非 ChatGPT | v2.0.1 Usability Probe 中观察用户行为 + v2.1.2 A/B 测试 | v2.0.1 → v2.1.2 | P0 |
| 用户因为"减少了一致性检查工作量"而持续使用 | 用户在管线内完成 3 章以上后的续写率 | v2.1.2+ | P1 |
| 方法论文本引导不会吓跑普通用户 | v2.0.1 中观察用户是否在画板交互中卡住 | v2.0.1 | P0 |
| 用户愿意为"更可控的 AI 输出"付费 | v2.1.2 A/B 测试中 B 组用户的付费意愿度 | v2.1.2 | P0 |

## 6. v2.0.1: Usability Probe

### 6.1 版本信息

| 字段 | 值 |
|------|-----|
| 版本号 | v2.0.1 |
| 目标 | **验证操作阻力（Usability）** ——验证"用户能不能用"，不验证"织梦机是否成立" |
| 依赖前置 | v2.0-H（主流程可用） |
| 风险等级 | 🟢 低风险 |
| 类型 | Usability Probe（可用性探针，非功能版本） |
| 核心原则 | 本版本不声称验证 PMF。本版本只回答：用户在不受方法论干扰的情况下，能否顺畅使用织梦机 |

### 6.2 版本定位

**这是织梦机的第一个"见到真实用户"的版本。**

v0.2 的评估报告指出的结构性问题——"4 个月方法论补完期间无用户反馈"——通过这个版本解决。在方法论骨架（v2.1.0）建设之前，先用一个轻量版本获取真实用户的操作数据。

**Usability Probe 解决的问题：**
- 用户会不会卡住
- 流程是否顺
- Driver 和实际 UI 是否一致
- AI 写入确认是否被理解
- 五画板管线在不强调方法论的情况下，用户能否自然走通

**Usability Probe 不解决的问题：**
- 不验证方法论的价值（那是 v2.1.2 的事）
- 不验证商业付费意愿（那是 v2.4 的事）
- 不验证 PMF（那是 v2.1.2 的决策判断）

### 6.3 核心范围（In Scope ≤ 5 项）

| # | 功能 | 优先级 | 说明 |
|---|------|-------|------|
| 1 | 一键速写入口 | P0 | 跳过方法论直接出正文。用户输入想法 → 一键生成草稿，不经过画板①→②→③→④→⑤ 的完整方法论管线。这是"低阻力入口"的核心 |
| 2 | 5 分钟路径 | P0 | 从启动到生成第一段正文在 5 分钟内完成。路径：打开应用 → 输入想法 → 点击"一键速写" → 看到正文 |
| 3 | Demo 项目 | P0 | 预置一个完整 Demo 项目供用户直接体验（含预设前提/结构/设定/两章正文），用户无需从零开始创建 |
| 4 | Markdown 导出 | P0 | 正文可导出为 .md 文件 |
| 5 | 反馈入口 | P0 | 内嵌反馈收集机制（问卷 + 行为日志），用于收集可用性数据 |

### 6.4 验收指标（非 PMF 指标）

本版本的所有验收指标都是**可用性指标**，不是**产品价值指标**。

| 指标 | 目标值 | 测量方法 | 是 PMF 指标？ |
|------|--------|---------|--------------|
| 5 分钟达标率 | ≥ 70% 用户可在 5 分钟内生成第一段正文 | 计时从启动到"一键速写"完成 | ❌ 这是可用性指标 |
| 用户卡住率 | ≤ 20% 用户在任意画板卡住超过 30 秒 | 行为日志 + 热力图 | ❌ 这是阻力指标 |
| 导出成功率 | ≥ 90% 用户可找到导出按钮并成功导出 | 行为日志 | ❌ 这是功能可用性 |
| 反馈提交率 | ≥ 30% 用户提交了反馈 | 反馈后台计数 | ❌ 这是参与度指标 |
| AI 写入确认理解率 | ≥ 80% 用户正确理解了 AI 输出三态的区别 | 弹出式小测验或后续访谈 | ❌ 这是 UI 传达有效性 |
| 管道完成率 | ≥ 50% 用户从输入想法走完了五画板管线 | 用户路径分析 | ⚠️ 部分反映可用性，不反映 PMF |

### 6.5.1 🔧 一键速写数据结构硬规则（v0.3.1 新增）

**规则：一键速写不得生成孤儿正文。**

一键速写的入口可以像猴子（低门槛、轻量、零术语），但底层不能像猴子——生成的正文数据必须是可转入正式五画板管线的结构化数据。

| 要求 | 说明 |
|------|------|
| **生成 QuickDraft** | 一键速写产出标记为 `QuickDraft` 类型，与正式管线正文结构相同但标记为"草稿" |
| **生成最小 PremiseCard** | 从用户输入自动推导最小前提卡（至少包含：一句话前提 + 类型判断），标记 `auto_inferred` |
| **生成最小 ChapterPacket/DraftPacket** | 每段正文生成对应的最小章节包（Layer① 写作契约 + Layer④ 执行层），后续可升级为完整版 |
| **可一键转入正式管线** | QuickDraft 完成后提供"转入正式管线"入口，触发：QuickDraft → 正式 PremiseCard → 各画板对应填充。用户确认后数据流向正式五画板 |
| **数据流不丢失** | 转入正式管线的过程中，用户已确认/修改的内容不得丢失 |

**原则表述：**

> 🔧 **入口可以像猴子，底层不能像猴子。** ——用户在一键速写界面看到的应该是"输入想法，点一下，就有东西出来"，但底层的每个字都必须是对应画板数据结构的一份合法实例。

### 7.4 方法论原则

| 原则 | 说明 |
|------|------|

### 9.4 评估方法

**🔧 A/B 测试设计（三组）：**

| 维度 | 设计 |
|------|------|
| **测试主题** | 同一故事想法（由测试组织者提供 3 个标准故事种子，用户选择 1 个） |
| **🔧 A1 组（对照组 1）** | 用户用 ChatGPT 自由裸聊生成第一章。无提示词协助，完全自由对话 |
| **🔧 A2 组（对照组 2）** | 用户用 ChatGPT + 标准提示词包生成第一章。提示词包包含写人/写景/写对话等通用模板，但不涉及织梦机方法论管线 |
| **🔧 B 组（实验组）** | 用户用织梦机方法论管线生成第一章（v2.1.0+v2.1.1 就绪的完整管线） |
| **🔧 用户数量** | 每组 ≥ 12 人，总计 ≥ 36 人 |
| **用户要求** | 有基本写作能力，非织梦机团队成员 |
| **盲测** | 评估者不知道哪组是 A1/A2/B（只看到"组1""组2""组3"的随机标签） |
| **控制变量** | 同一故事种子、同一 AI 模型（三组均使用 GPT-4 或同等模型） |

### 9.5 A/B 测试流程

```
① 招募测试用户（≥36人）
② 随机分组（A1组 / A2组 / B组）
③ 各组分发同一故事种子（3选1）
④ A1组：用户在 ChatGPT 中用自由对话生成第一章
⑤ A2组：用户在 ChatGPT 中用标准提示词包生成第一章
⑥ B 组：用户在织梦机中用方法论管线生成第一章
⑦ 收集生成结果（正文 + 用户反馈 + 行为日志）
⑧ 专家组盲评（不知道组别）
⑨ 用户自评问卷
⑩ 统计分析 → 结果报告
⑪ Go/No-Go 决策（织梦机是否进入 v2.2 商业化准备）
```

### 9.6 评估矩阵

| 维度 | 测量方法 | 判定标准 | 计量方式 |
|------|---------|---------|---------|
| **设定一致性** | 生成结果是否符合故事设定（角色不崩、规则不违反） | 5 分制专家评分（多专家取均值） | A 组均值 vs B 组均值，t-test |
| **角色稳定性** | 角色行为是否前后一致，性格是否保持 | 5 分制专家评分 | A 组均值 vs B 组均值 |
| **章节结构** | 章节推进是否有结构感、节奏感 | 5 分制专家评分 | A 组均值 vs B 组均值 |
| **用户续写意愿** | "你是否愿意继续写第二章？" | 5 分制 Likert | A 组均值 vs B 组均值 |
| **优点辨别** | 用户能否说出"织梦机比 ChatGPT 好在哪里" | 开放式回答 → 主题编码 | B 组回答聚合 |
| **易对比** | "哪个更容易继续写第二章？" | 5 分制 | 直接对比 |

**🔧 Go/No-Go 标准：**

| 结果 | 含义 | 决策 |
|------|------|------|
| 🔧 B 组在 ≥ 3 个维度同时优于 A1 和 A2（p < 0.05） | 织梦机价值验证通过——赢的不只是裸聊，连标准提示词都赢 | **Go** → 进入 v2.2 |
| 🔧 B 组优于 A1（裸聊）但未优于 A2（标准提示词包） | 织梦机比裸聊好，但标准提示词就能追平差距 | **Go with gaps** → 优化方法论引导 → 研究 A2 组提示词中的有效内容并内化到织梦机 |
| 🔧 B 组仅与 A1 持平，A2 最优 | 裸聊和织梦机一个水平，标准提示词最有效 | **No-Go with analysis** → 需重新评估织梦机方法论相对于标准提示词包的增益 |
| 🔧 B 组无显著差异 | 方法论管线未产生可感知价值 | **No-Go with analysis** → 分析原因 → 暂停或重设计 |
| 🔧 A1 或 A2 显著优于 B 组 | 织梦机减损而非增值 | **No-Go** → 退回到产品定义阶段 |

### 19.3 🔧 定价原则（v0.3.1 修正）

- **方法论隐性内嵌**：方法论溢价不单独列出，内嵌于点数价格中
- **免费路径完整**：免费额度应足够用户体验完整的"一句话→第一章正文"管线
- **🔧 BYOK 不消耗模型托管点数**：用户自带 Key 后，模型调用成本由用户自行承担。BYOK 用户不消耗平台的模型点数，但可消耗平台功能点 / 高级功能额度（如导出、模板、七诊分析等非模型调用的平台功能）
- **🔧 托管模型才消耗模型点数**：使用平台共享 API Key 的托管模式消耗模型点数。两种消费路径并行不悖——BYOK 用户免模型费但可以选择购买平台功能额度
- **🔧 原则表述：** BYOK = 自带车辆（模型），不付油费（模型调用费），但付过路费（平台服务费）

## 21. 附录：Usability Probe 与 Value Probe 的区别说明

### 21.1 为什么需要两个探针

**这是一个在 v0.2 中被错误合并的关键区别。**

之前的版本路线有一个隐含假设：如果用户能用织梦机顺畅地写出第一章，就说明织梦机的产品价值成立了。这个假设是错误的。

**可用性（Usability）和价值（Value）是两个完全不同的验证问题：**

| 维度 | Usability Probe（v2.0.1） | Value Probe / PMF Probe（v2.1.2） |
|------|--------------------------|----------------------------------|
| **问的问题** | "用户能不能用？" | "用了是否比不用好？" |
| **验证对象** | 操作阻力（R） | 产品价值（T × A × B × η） |
| **失败含义** | UI 设计问题、交互流程问题、学习成本问题 | 产品定位问题、方法论价值不成立 |
| **失败后可修复** | 改 UI、改流程、改文案——**可修复** | 需要重新评估产品方向——**需要大改或放弃** |
| **测试方法** | 行为日志 + 问卷 + 面谈 | A/B 测试 + 盲评 + 统计显著 |
| **需要方法论就绪？** | **不需要。** v2.0.1 的"一键速写"跳过方法论 | **需要。** v2.1.2 的 B 组使用完整方法论管线 |
| **用户数量** | ≥ 20（定性 + 定量） | 🔧 ≥ 36（三组分组定量，每组≥12人） |
| **结论类型** | "这里用户容易卡住，改" | 🔧 "织梦机管线 vs ChatGPT 裸聊 vs ChatGPT 标准提示词包——哪一条路径产出更好" |

### 21.2 两探针在版本路线中的位置

```
v2.0-H: 主流程可用
  ↓
v2.0.1: 可用性探针 ─────────────→ 回答"用户能不能用"
  ↓                                      ↓
v2.1.0: 方法论骨架                    修复操作阻力
  ↓
v2.1.1: 方法论集成
  ↓
v2.1.2: 价值探针 ─────────────→ 回答"用了是否比不用好"
  ↓                                      ↓
v2.2: 智能层                         Go → 继续 / No-Go → 重新评估
```

### 21.3 为什么不合并为一个版本

| 方案 | 问题 |
|------|------|
| 先做方法论 → 再做探针 | 方法论文本（PremiseCard v2 五步、麻雀模式等）本身会增加操作阻力。如果在方法论就绪后才做可用性验证，无法区分"方法论引导带来的阻力"和"UI 本身的阻力" |
| 先做探针 → 再做方法论 | 只有可用性（UI 是否顺畅）无法回答"方法论是否创造了价值"——用户能顺畅使用不等于方法论有价值 |
| 两探针合并 | 无法分辨失败原因是"UI 难用"还是"方法论没有价值"。两个变量混在一起，结论无意义 |

**结论：两探针必须分离。v2.0.1 先排除"UI 阻力"变量，v2.1.2 再单独验证"方法论价值"变量。**

### 21.4 两探针之间的数据流向

```
v2.0.1 Usability Probe 产出:
  ├── 操作阻力清单（按严重程度排序）
  ├── 修复建议（按工作量评估）
  ├── 用户行为热力图（各画板使用频率/卡住时间）
  └── "待方法论验证假设"（用户行为中暗示方法论可能有价值/无价值的迹象）
                              │
                              ▼
                   → 输入到 v2.1.0 方法论骨架设计
                   → 输入到 v2.1.2 A/B 测试假设设计
```
