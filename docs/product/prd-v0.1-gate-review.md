## 四大审查维度

### 2.1 Scope Drift（范围偏移）

**整体判断：方向一致，存在三处可追踪的范围偏移。**

#### 与原始 PRD 的一致性

| 维度 | 结论 | 证据 |
|------|------|------|
| 产品定义的一致性 | **一致** | v2.0 的"五画板管线 + AI 输出三态 + 数据依赖链"均在 B/C/D 三轮中验证通过 |
| 五画板核心交互 | **一致** | 画板①→⑤ 的管线骨架在 D 轮完成后已完整就绪 |
| AI 交互模型 | **一致** | AI 输出三态（discuss/suggest/write_preview）已在 D 轮实现 |
| Phase 4 拆分 | **方向正确，但与原 PRD 表述不同** | 原 PRD 将 reverse pipeline/scoring/BYOK/mobile 统一打包为 Phase 4 (v2.1+)。v0.1 将其拆解为 v2.3-v2.6。这是合理的细化，但需要显式标注明细 |

#### 范围漂移风险 #1：画板④三档细度模式 — AC 遗漏

原始 PRD 第 3.5 节将"三档细度模式（速写/标准/精编）"列为验收标准，但在 B/C/D 三轮实现中从未触及。D 轮仅实现了审核模式/完整模式的双模式切换。这是一个 P0 级 AC 的沉默丢失。

**建议：** 在 v0.2 中明确三档细度模式的归属版本（推荐 v2.1 或明确降级为 P1）。

#### 范围漂移风险 #2：Phase 2 vs Phase 3 交换未被记录

原始 PRD 的路线：
```
Phase 2 (质量层：脚本Batch B/七诊LLM/咬合检查/写作契约更新) → Phase 3 (方法论增强：八体/天/地/人展开/完整十二时位)
```

v0.1 新路线：
```
v2.1 (方法论补完：画板①完整五步/画板②地图缩放/画板③方法论补完) → v2.2 (智能层：七诊/八体/知识边界检测)
```

实际发生了方法论补完与质量层的顺序交换。这个交换**是合理的**（方法论应先于质量增强），但不应沉默发生。需要在 v0.2 中说明这个调整及其理由。

#### D0 Scope Freeze 边界

D 轮的 Out of Scope 列表在 v0.1 版本路线中得到尊重：完整画板①方法论、画板②地图缩放、画板③方法论补完均推迟到 v2.1+。一致。

### 2.2 Intent Mismatch（意图错配）

**整体判断：北极星对齐，商业节奏有一个待决缺口。**

#### 与北极星的一致性

北极星定义："一句话前提 → AI 走完 5 画板管线 → 用户在每个输出节点确认/修改 → 成文"

v2.0（D 轮完成状态）已实现这个北极星的基础版本。后续版本的规划（v2.1 方法论增强 → v2.2 质量层 → v2.3 反向管道）均在扩展这个北极星的可达范围，而非偏移。

#### 核心原则贯彻情况

| 原则 | 对齐状态 | 评估 |
|------|---------|------|
| 定价先于漫剧 | **对齐** | 定价 UI (v2.3)，漫剧模板 (v2.5) |
| 商业化先于模板 | **对齐** | 商业化 (v2.4)，模板 (v2.5) |
| 模板先于平台 | **对齐** | 模板 (v2.5)，平台 (v2.6) |

**意图错配风险 #1：定价决策未分配版本**

核心原则"定价先于漫剧"要求定价决定做出后，才能规划漫剧。但当前规划中：

- 定价 UI / Cost Meter → v2.3
- 定价方案决策（具体点数、免费额度策略）→ **未分配至任何版本**

这产生了一个断点：v2.3 开始开发定价 UI 时，定价方案可能尚未确定，导致 UI 开发方向不确定。

**建议：** 将定价决策分配至 v2.2，确保 v2.3 开始时有确定的定价方案。

#### 前置依赖链检查

| 依赖 | 状态 | 评估 |
|------|------|------|
| 知识边界检测器 (v2.2) 依赖 AI 三态写入 (v2.0) | **合理** | D 轮已实现三态写入 |
| 八体风格 (v2.2) 依赖正文管线完整 (v2.0) | **合理** | C 轮已完成 |
| 反向管道 (v2.3) 依赖正向管线稳定 (v2.0) | **合理** | v2.0 已完成 |
| 反向管道 (v2.3) 依赖方法论补完 (v2.1) | **合理但可能有风险** | 如果 v2.1 方法论未完成，反向管道的设定提取质量会下降 |

路径上**不存在断裂的跨版本依赖**。所有依赖链是线性的、前置版本完成后即可进入后置版本。

### 2.3 Internal Consistency（内部一致性）

**整体判断：版本号体系存在未定义映射，版本间依赖链合理。**

#### 版本号体系问题

当前系统中存在两套命名体系：

| 体系 | 使用范围 | 值域 |
|------|---------|------|
| Phase (工程编号) | 原始 PRD / 实施计划 | Phase 0, 1A, 1B, 1C, 2, 3, 4 |
| v2.x (产品版本号) | D0 文件 / D 轮报告 / v0.1 路线 | v2.0, v2.1, v2.2, v2.3, v2.4, v2.5, v2.6 |

从 D 轮报告推断的映射关系：
```
Phase 0 + 1A + 1B + 1C = v2.0 (已完成)
Phase 2 (质量层) 主体 → v2.1 方法论补完 (范围已大幅调整)
Phase 3 (方法论增强) 主体 → v2.2 智能增强层
Phase 4 (扩展层) → v2.3 反向管道 + v2.4 商业化 + v2.5 模板 + v2.6 平台
```

**问题：** 这个映射从未被正式记录或批准。原始 PRD 的 Phase 和新的 v2.x 之间存在范围调整和顺序交换，但没有一个文档说明这些调整的理由和影响。

**必须解决：** 在进入 v2.1 设计前，必须明确定义两套编号的等价关系或分叉理由。

#### 版本间依赖链

```
v2.0 (稳定基线)
  → v2.1 (方法论补完：画板①②③完整交互)
     → v2.2 (质量层：七诊/八体/知识边界检测)
        → v2.3 (反向管道：导入/分析/预填)
           → v2.4 (商业化：定价 UI/Cost Meter/发布工具)
              → v2.5 (垂直模板：漫剧模板)
                 → v2.6 (平台：交易系统)
```

**依赖链是合理的。** 每个版本的输入依赖前序版本的核心能力就绪。没有循环依赖。

#### "不做清单"的充分性

D0 文件的第 4 节 (Out of Scope) 提供了 D 轮的明确不做清单。但 PRD v0.1 本身缺少每个版本的独立"不做清单"。这个缺口与产品经理的 P1-2 一致。

#### 我对产品经理审查的额外发现

**发现 A (可操作性问题)：v2.1 的拆分粒度不对**

产品经理建议 v2.1a/b/c 按画板拆分（v2.1a = 画板②地图缩放，v2.1b = 画板①+③方法论，v2.1c = 收尾）。但按画板拆分可能导致跨版本的上下文切换成本：
- v2.1a 做画板②，v2.1b 做画板①+③ → 设计上下文在 v2.1a 和 v2.1b 之间切换

**建议替代方案：** 按"功能簇"拆分
- **v2.1a (方法论核心)：** 画板①完整五步 (规格已就绪，风险低) + 画板③方法论 (麻雀模式/三层角色卡/势力卡片) — 这两个画板是方法论完整度的核心，规格相对确定
- **v2.1b (结构交互)：** 画板②地图缩放 L1-L4 + 三项决策 — 最高风险，独立版本降低连带伤害
- **v2.1c (方法论收尾)：** 画板③剩余 (咬合检查/天/地/人展开) + 三档细度模式修复

这样 v2.1a 可以先启动（低风险），v2.1b 独立攻关（高风险不阻塞其他），v2.1c 收尾。

**发现 B (方法论文档尚未注册为 AI skill)**

原始 PRD 第 7 节列出了 6 个 Skills（S-1 到 S-6）：世界观构件、人物屠龙术、八字六变、十二时位、七诊、八体。这些是织梦机的核心方法论壁垒。当前只有 S-1 和 S-2 的部分内容在 C/D 轮中被 AI 调用。S-3 到 S-6 的完整注册和激活尚未完成。

这影响 v2.1 (方法论补完) 的交付：如果方法论没有注册为 AI skill，用户在画板中看不到方法论引导，方法论壁垒停留在"文档层面"。

**建议：** 在 v2.1 启动时将 6 个 Skills 的完整注册作为一个前置任务。

**发现 C (BYOK 策略未定)**

v1.3 已有 BYOK。v2.0-v2.6 规划中没有说明 BYOK 的状态变化：
- v2.0 的 BYOK 是否与 v1.3 兼容？
- v2.2 如果引入托管额度，BYOK 与托管额度的关系是什么？
- 如果 BYOK 作为免费用户方案保留，它的商业化转化路径是什么？

这虽然不是立即阻塞，但到 v2.4 (商业化) 时必须明确。

#### 产品经理与丞相讨论分歧中的立场

| 讨论点 | 产品经理立场 | 我的立场 | 理由 |
|--------|------------|---------|------|
| v2.1 拆分 | 拆为 v2.1a/b/c | **同意拆分，建议按功能簇而非按画板切分** | 见"发现 A" |
| 画板②优先级 | 画板②（最高风险）优先 | **同意画板②优先级应高，但不建议独占 v2.1a** | 画板②是最高风险的核心交互，但不应让低风险的画板①③等待；分轨并行更优 |
| 漫剧子产品线 | 保持模板级别，不提前做子产品线决策 | **同意** | v2.5 之前的判断缺乏数据支撑 |
| BYOK 策略 | 互补策略（BYOK 免费基线 + 托管额度付费） | **同意方向，但建议 v0.2 中记录框架** | 避免 v2.4 时仓促决策 |
| 反向管道时机 | 保持 v2.3，不提前 | **同意** | 验证了正向管线的可靠性后才能做反向 |
| 方法论优先于智能检测 | 方法论优先 | **同意** | 方法论是核心壁垒，智能层是增强层。排序正确 |

## 阻断与条件

### 进入下一阶段的条件（Gate Conditions）

| # | 条件 | 严重度 | 解决时限 | 说明 |
|---|------|-------|---------|------|
| C-1 | **明确定义 Phase 与 v2.x 的等价映射，或在 v0.2 中正式分叉** | **必须** | 进入设计/架构前 | 这是版本编号体系的最终澄清。不解决则团队无法对齐版本边界 |

### 不通过条件（我会 FAIL 的情况）

如果以下情况发生，我会将决策从 PASS_WITH_NOTES 升级为 FAIL：
1. 进入下一阶段时 C-1 或 C-2 未被处理
2. v2.1 进入实现阶段时 scope 未拆分
3. 发现版本间出现未声明的循环依赖

## 最终裁决与建议动作

### Gate Decision: **PASS_WITH_NOTES**

### 裁决理由

PRD v0.1 的产品方向和版本路线正确、逻辑合理，与 Owner 的北极星和核心原则高度一致。v2.0 已通过 B+C+D 三轮验证，证明管线基础可运行。扩展至 v2.0-v2.6 的七版路线是自然的增量扩展，没有方向性错误。

但版本路线规划的细节（编号映射、scope 校准、商业节奏缺口）存在两个必须解决的问题。这些问题不影响"产品方向对不对"的判断，但会影响"团队能不能对齐执行"。

### 建议动作

1. **短期（v0.2 更新前）：**
   - 明确定义 Phase 与 v2.x 的映射关系（C-1）
   - 决定 v2.1 的拆分方案（C-2）
   - 将定价决策分配至 v2.2（C-3）
   - 澄清画板④三档模式 AC 状态（C-4）
   - 在 v0.2 中增加每个版本的"版本完成清单"和"不做清单"

2. **中期（v2.1 开始前）：**
   - 注册 6 个方法论 Skills（S-1 到 S-6）到 `.claude/skills/zhimengji/`
   - 明确 BYOK 在 v2 中的策略
   - 更新 `CONTEXT.md` 中的术语表，统一"八体"为"八体风格"

3. **长期追踪：**
   - 监控 v2.2 (质量层) 的工作量估算，Phase 2 原估 4 周可能不足
   - 画板②地图缩放交互的高风险需单独标记和缓解计划

### Return To

无需退回。PASS_WITH_NOTES 允许进入下一阶段（设计/架构），但必须携带上述 4 个条件。
