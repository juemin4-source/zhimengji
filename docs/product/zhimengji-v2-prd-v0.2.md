## 1. 产品北极星

### 1.1 一句话定义

**织梦机是一款 AI 管线化小说创作工具——用户和 AI 在同一块画板上工作，AI 走管线，用户在每站有否决权。**

它不是"AI 全自动写作工具"（Notion AI / Sudowrite），也不是"纯手工编辑器"（Scrivener / Ulysses）。它是两者之间的一条结构化管道：作者像导演，AI 是他的创作班子。AI 按画板顺序逐一输出结构化的创作素材，用户在每一站确认、修改或推翻。

### 1.2 最小闭环

> 一条前提句 → AI 走完 5 画板管线 → 用户在每个输出节点确认/修改 → 成文。

织梦机的最小可交付价值不是某个按钮或某个面板，而是这条管线的**端到端可运行性**。从模糊想法到成品正文，用户只需做"确认/修改"的轻量操作，重活（生成前提句、展开结构、填充设定、拆解细纲、撰写正文）由 AI 在画板约束下完成。

### 1.3 三大核心痛点

| 痛点 | AI 写作工具的现状 | 织梦机的解法 |
|------|------------------|-------------|
| **上下文丢失** | AI 对话窗口越长越乱，前文角色/设定在 10 轮对话后开始混淆 | 四层自包含细纲包：每章压缩为"写作契约 + 设定快照 + 压缩叙事 + 执行层"的完整包，任意 AI 拿到即可正确写作 |
| **幻觉/不可控** | AI 擅自创建角色、使用未设定规则、写出"角色知道不应该知道的信息" | 知识边界检测器 + 临时假设机制：AI 遇到设定空白只能标记为"临时假设"，不得静默补全 |
| **方法论缺失** | 竞品只提供"帮我写"一个按钮，没有从想法到成文的结构化方法论引导 | 五画板管线 + 12 个方法论技能（八字六变/十二时位/世界构件/人物屠龙术/七诊/八体），用户在整个创作过程中被方法论赋能而非替代 |

### 1.4 生态位定位：两个维度的无人区

织梦机的生态位不能用"功能对比表"来描述——因为它本质上不是"又一个 AI 写作工具"。它的竞争对手不是 Novelcrafter 或 Dabble，而是"空白页面带来的创作焦虑"和"AI 管线的上下文丢失"。要理解这个生态位，需要看两个维度：

**维度一：方法论深度**——产品是否内置了一套可执行的创作方法论。
**维度二：AI 管线化程度**——产品是否将创作流程从想法到正文组织为结构化的 AI 驱动管道。

```
                          方法论深度（高）
                               │
                               │
               World Anvil     │     织梦机 ⭐
               （设定有方法论  │     （方法论 x 管线 = 无人区）
                但无 AI 管线） │    
                               │
  ────────────────┼─────────────────────────── AI 管线化程度
                               │
       Scrivener / Ulysses     │     Novelcrafter
       （纯手工，无 AI）       │     （有 AI 但无方法论）
                               │     Dabble
                               │     （轻 AI，无方法论）
                               │
                          方法论深度（低）
```

### 1.5 六维差异化壁垒

| 维度 | 织梦机 | 竞品现状 | 护城河类型 |
|------|--------|---------|-----------|
| **自有方法论** | 八字六变、十二时位、七诊、八体、世界观构件表、人物屠龙术——6 套方法论全部可执行为 AI 技能 | 无自有方法论；Novelcrafter/Dabble 仅提供通用 AI 生成 | IP 壁垒（Owner 独家，不可逆） |
| **画板管线** | 5 个画板构成数据强依赖链，AI 严格按管线走 | Novelcrafter Beat sheet 弱结构；Dabble Storyboard 仅时间线；Scrivener Corkboard 通用卡片 | 产品架构壁垒（工程投入大） |
| **四层细纲包** | 自包含 + 可移植：导出纯文本后任意 AI 可执行 | Novelcrafter Active Context 依赖 AI 记忆；Dabble 无细纲系统 | 产品设计壁垒（细粒度专利性） |
| **风格自适应** | 八体 x 10 维量化空间，AI 隐性保持用户风格一致 | Novelcrafter 需手动角色卡；Dabble 手动风格设定 | 方法论文献壁垒（文心雕龙本身不可复制） |
| **反向管道** | 导入完稿 → 反向分析全部画板 + 评分 | 无竞品提供结构化反向分析 | 先发壁垒（管线反向利用需正向管线已存在） |
| **检测脚本** | 本地规则脚本 + LLM tool 双模引擎 | 无竞品提供系统性文本质量检测 | 工程壁垒（双模引擎的集成成本） |

### 1.6 核心设计原则

| 原则 | 含义 | 影响 |
|------|------|------|
| **前期约束，后期自由** | 画板①→②→③严格串行不可跳过；画板④→⑤推荐串行但允许宽松 | 数据依赖链驱动 UI 导航：未就绪画板可点击查看说明，但不可编辑 |
| **追问驱动，非填空驱动** | 每个画板的交互核心是"问题"而非"字段" | 用户看到的是自然语言问题，不是字段名。例如："这个世界是什么颜色？"而非"美学辨识度：[输入]" |
| **AI 是填槽助手，不是创作主体** | AI 读取上一阶段输出，填当前阶段的"槽" | AI 的产出必须标记"AI 生成"，用户可以逐项确认/修改/拒绝 |
| **四层自包含** | 每章细纲打包为自包含包，不依赖 AI 上下文记忆 | 导出到任意 AI 工具（DeepSeek / ChatGPT）都能写出正确正文 |
| **可逆设计** | 正向：想法→正文；反向：正文→分析全部画板 | 反向数据标记 source 来源，所有分析结果可修改 |

## 4. 旧 Phase deprecated 说明与对照表

### 4.1 Deprecated 声明

**旧 Phase 编号（Phase 0 / 1A / 1B / 1C / 2 / 3 / 4）已废弃。**

原因：
- 旧 Phase 体系与产品版本号 v2.x 之间存在未定义的映射关系，导致团队无法对齐版本边界
- Phase 4 (v2.1+) 打包了过多功能，与实际交付节奏脱节
- 旧 Phase 的范围在迭代过程中发生了隐式调整（如 Phase 2 与 Phase 3 的顺序交换），但没有文档记录

**从即日起：**
- 后续主路线只使用产品版本号：v2.0 → v2.1.0 → v2.1.1 → v2.1.2 → v2.2 → v2.3 → v2.4 → v2.5 → v2.6
- 旧 Phase 仅作为附录对照表，不作为团队排期语言
- 任何排期/计划文档中不得出现 Phase 编号

### 4.2 隐式调整记录

在从旧 Phase 体系过渡到新版本体系过程中，发生了以下隐式调整，在此显式记录：

| 调整 | 旧体系 | 新体系 | 理由 |
|------|--------|-------|------|
| Phase 2 与 Phase 3 顺序交换 | Phase 2（质量层）→ Phase 3（方法论增强） | v2.1（方法论补完）→ v2.2（智能检测层） | 方法论应先于质量增强——没有方法论，智能检测缺乏语境 |

| v2.1 拆分 | v2.1 一个版本 | v2.1.0/v2.1.1/v2.1.2 三个子版本 | 画板②地图缩放是最高风险项，需独立验证 |

### 6.2 风险说明

画板②地图缩放（@xyflow 深度集成 + L1-L4 四层分形树 + 三项决策嵌入层级）是整个织梦机产品中**最高风险的核心交互**。

风险来源：
- **交互范式未验证**：地图缩放用于叙事结构图的交互范式在竞品中未被验证过。Google Maps 式 zoom 在内容创作工具中的用户体验是否成立，目前是假设
- **@xyflow 集成复杂度**：需要在 @xyflow 上实现"四层分形树状结构 + 每层可缩放 + 层级间状态保持"的交互，超出了 @xyflow 的开箱能力
- **三项决策嵌入层级**：将决策 UI 嵌入到地图缩放的每一层中，是全新的交互设计——每层显示什么决策，决策如何影响结构，需要在真实 UI 中验证

**缓解措施：**
- 本版本定义为技术 Spike，不做完整产品化
- 不做画板①③ 的重做
- 产出物是"可运行的交互原型"，不是"交付给用户的完整功能"
- 经过 Spike 验证后，交互方案可能根据发现的问题调整

### 6.3 核心范围

### 18.2 v2.2 必须做出的定价决策

| 决策项 | 默认方案 | 需要 Owner 确认 |
|--------|---------|----------------|
| **免费额度策略** | 每日 xx 点（具体数额），免费用户可走完"一句话→第一章正文"完整路径 | ✅ |
| **点数单价** | 生成前提句=1点，写一章=5点，其他操作依此类推 | ✅ |

## 19. 画板④三档细度模式处理结论

### 19.1 状态

**原始 PRD 中画板④三档细度模式（速写/标准/精编）被列为验收标准，但在 B/C/D 三轮中从未实现。D 轮只实现了审核/完整双模式。**

**处理结论：** 三档细度模式推迟到 v2.1.2 或 v2.2，不作为 v2.0 Release Gate。

### 19.2 决策理由

1. **优先级不足**：三档细度模式是交互增强，不是管线核心功能。管线核心（五画板骨架 + AI 点火 + 控制力）已在 v2.0 完成
2. **D 轮审核模式已解决主要痛点**：D 轮实现的审核/完整双模式已经解决了"字段过多导致窒息"的核心问题
3. **等待方法论就绪**：三档模式的精编模式需要画板①③ 的完整方法论就绪（v2.1.1）后才能充分利用
4. **由 v2.1.2 集成层实现更合理**：三档模式涉及画板④与上下游联动，放在 v2.1.2（集成层）中实现更加合理

### 19.3 三档模式规划（v2.1.2 或 v2.2）

| 模式 | 适用用户 | AI 行为 | 用户审核 |
|------|---------|---------|---------|
| **速写** | 猴子、快速推进 | 自动生成，默认折叠 Layer ④ | 可选跳过审核 |
| **标准** | 大多数用户 | 自动生成，Layer ④ 可展开 | 审核 Layer ③ |
| **精编** | 控制型作者 | 自动生成后用户精细调整全部字段 | 调整 Layer ③+④ |

以下功能经过讨论已明确**不在织梦机 v2 任何版本中实现**：
