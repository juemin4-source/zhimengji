# 竞品调研：Obsidian Longform 插件

## 概述

Obsidian Longform 是 Kevin Barrett 开发的免费开源 Obsidian 社区插件，专门用于管理长篇写作项目（小说、剧本、学术论文等）。它通过将多个 Markdown 笔记组织成有序的"场景"（Scenes）来构建完整的"手稿"（Manuscript），并提供版本管理、字数统计、编译导出等写作流程工具。

Longform 插件目前版本为 v2.0.3（截至 2024 年），成熟度约 5 年，社区活跃度较高但有大量未关闭 issue（100+）。它被 Obsidian 用户广泛用于替代 Scrivener 等商业写作软件。

---

## 核心功能分析

### 1. 项目结构管理

Longform 的核心创新是将 Obsidian 的文件夹结构映射为长篇写作项目的层次结构。

**项目类型：**
- **多场景项目（Multi-scene）**：一个文件夹中的多个 `.md` 文件作为场景，按顺序组合成手稿
- **单场景项目（Single-scene）**：单个笔记文件，适合短篇故事或随笔

**索引文件（Index File）：**
Longform 在项目文件夹中生成 `index.md` 文件，通过 YAML frontmatter 跟踪所有场景的元数据：

```yaml
longform:
  format: scenes
  title: My Great Novel
  draftTitle: Draft 1
  sceneFolder: /
  scenes:
    - Chapter One - The Beginning
    - - Chapter Two Scene A
      - Chapter Two Scene B
    - Conclusion
  ignoredFiles:
    - notes
    - "*-scratch"
  workflow: Default
```

关键设计原则：**Longform 从不修改笔记内容**——它只重写索引文件。这意味着它不会意外删除或丢失任何写作内容。

### 2. 场景（Scenes）与章节组织

| 能力 | 说明 |
|------|------|
| **场景列表** | 项目侧栏显示所有场景的有序列表，支持拖拽排序 |
| **嵌套层级** | 场景可嵌套缩进，创建 章节 → 场景 的层级（用 YAML 数组的嵌套表示） |
| **独立文件** | 每个场景是一个独立的 Markdown 文件，保持文件小巧聚焦 |
| **快速切换** | 侧栏点击场景即可跳转编辑，无需在文件管理器中寻找 |
| **忽略规则** | 可在索引中配置忽略文件模式，不列入场景列表 |

### 3. 写作与版本管理

**多草稿（Drafts）：**
Longform 支持为同一项目创建多个草稿版本，每个草稿都有自己的场景列表和字数统计。用户可以在不同草稿之间切换，比较版本差异。

**字数追踪：**
- 场景级别字数统计
- 草稿级别字数统计
- 项目累计字数统计
- 每日写作目标设置和进度追踪

**专用写作侧栏：**
- 收集 vault 中所有 Longform 项目
- 快速切换项目和草稿
- 显示字数进度和写作目标

### 4. 编译/导出工作流

Longform 的编译引擎是它的核心差异化功能。

**编译流程：**
1. 用户选择编译工作流（Workflow）
2. 插件将所有场景按顺序合并为单个 Markdown 文件
3. 应用格式规则（移除 wikilink、外部链接等）
4. 输出合并后的手稿文件

**典型导出链：**
```
Longform 编译 → 合并 Markdown → Pandoc 插件 → DOCX/ePub/PDF
```

Longform 本身只输出 Markdown。要获得出版格式，通常需要串联 Pandoc 或其他工具。这是它和 Scrivener 的重要差距。

**已知痛点：**
- 场景排序偶尔漂移（该 issue 已被报告超过 100 次）
- 编译按钮偶发不可点击，需要重启
- Pandoc 导出时 YAML frontmatter 可能丢失
- 没有原生 DOCX/ePub 导出

### 5. 自定义与扩展

- **Custom CSS**：`.longform-leaf` class 允许用户为写作环境单独设置样式（如衬线字体、暖色背景）
- **Compile Workflow 可编程**：支持社区贡献的自定义编译步骤
- **与其他插件配合**：典型搭配为 Dataview（查询）、Templater（模板）、Pandoc Plugin（导出）

### 6. 社区评价

**优势：**
- 免费（对比 Scrivener $60）
- 纯文本 + Markdown，未来可迁移
- 不修改笔记内容，安全可靠
- 与 Obsidian 生态无缝集成

**劣势：**
- 没有实时协作能力
- 导出流程复杂（依赖第三方工具）
- 场景排序偶发 bug
- 没有内置的"大纲视图"——依赖 Obsidian 本身的 Graph View 或其他插件

## 对织梦机的启发

### 1. 索引文件模式值得借鉴

Longform 用 YAML frontmatter 做场景注册表，不修改笔记内容的设计非常优雅。织梦机的"章节管理"也可采用类似模式——一个项目索引文件记录所有章节/场景的元数据（顺序、状态、字数、关联对象），而不侵入叙事单元本身的内容。

### 2. 场景级元数据追踪

Longform 的场景级别字数统计是基本的，但织梦机可以做得更深：
- 每个叙事单元关联的角色/设定数量
- 单元内部的 WikiLink 引用密度
- 单元的情感曲线标签
- 单元的一致性评分（由 Skill 引擎产生）

### 3. 编译流程的思路

Longform 的 workflow 编译是一个可扩展的架构。织梦机的"导出"功能可以参考这个模式，但更进一步：
- 支持多种导出目标（markdown 合并、PDF 章节书、结构化 JSON）
- 编译过程中嵌入 AI 处理步骤（风格统一检查、一致性修正）
- 编译前自动运行一致性检查，标记问题

### 4. 多草稿版本管理

Longform 的草稿系统相对简单（以项目为单位）。织梦机可以做得更好：
- 按叙事单元级别的版本管理（而非整个项目）
- AI 辅助的版本对比（自动标注两版之间的叙事差异）
- 分支式写作实验（类似 git branch，但面向叙事）

### 5. 非破坏性编辑

Longform "不修改笔记内容"的设计理念是织梦机必须坚持的：
- 索引数据与内容数据严格分离
- AI 修改必须通过"变更记录"呈现，用户接受后才写入
- 任何自动化的结构性调整都应可回滚

## 差异化建议

### 织梦机不应该简单复制的功能

| 功能 | 原因 |
|------|------|
| YAML frontmatter 式索引 | 织梦机已有 SQLite 结构化存储，不需要平铺在 frontmatter 中 |
| 纯 Markdown 文件作为场景单元 | 织梦机使用 Tiptap 富文本 + 结构化对象，不是纯文件系统组织 |
| 编译为单个 Markdown | 织梦机的输出目标应该更多元（按角色提取章节、按情感线组织等） |
| 依赖 Pandoc 导出 | 织梦机应该内置导出引擎，不依赖外部工具链 |

### 织梦机的差异化方向

1. **智能场景管理**：不仅仅是排序，而是 AI 辅助的场景拆分/合并建议，基于叙事密度和节奏分析的章节划分

2. **叙事结构可视化**：Longform 只有列表视图，织梦机的画板可以展示场景关系图、时间线、角色出场密度图

3. **AI 驱动的版本迭代**：不只是"保存不同版本"，而是让 AI 分析版本之间的叙事差异（"这一版第二章的紧张度降低了 30%"）

4. **场景级诊断**：每个场景在编写过程中持续接收 Craft Skill 诊断（节奏、一致性、类型承诺），无需等待编译后才发现问题

5. **不是"文件管理器 + 插件"**：织梦机是专为叙事构建的应用，Longform 是在通用笔记系统上搭建的写作工具。织梦机可以把叙事结构作为一等公民，而不是文件系统的上层抽象

6. **导出即出版物**：织梦机应内置排版引擎，直接从结构化数据导出排版就绪的 PDF/ePub，而不是让用户自己串联 Pandoc

---

## 总结

Longform 是 Obsidian 生态中最成熟的长篇写作工具，它的非破坏性索引设计、场景级管理、编译工作流值得织梦机参考。但它本质上是在通用笔记系统上叠加写作功能——织梦机作为专为叙事构建的应用，在前述差异化方向上存在显著的机会空间。

最大的启发是：**将章节/场景管理从文件组织提升为叙事结构的第一等公民**，并围绕这种结构化数据构建 AI 驱动的诊断、分析和导出能力。
