## 1. 插件愿景（一句话）

> **织梦机的插件，是第三方作者和开发者向织梦机注册新"创作能力"的机制——方法论、AI技能、输出格式、数据源，都可以通过插件扩展。织梦机核心管线锁定，外围可扩展。**

[推测] 这句话的核心判断是：织梦机的插件不是"代码扩展"，而是"创作能力包"。插件的内容面向创作场景，而非面向软件工程场景。

## 2. 用户场景

### 场景 A：方法论作者注册新画板流程

[推测] 一位资深的类型小说作家使用织梦机后，认为现有的"五画板管线"不适合推理小说的创作流程。她编写了一个"推理小说专属画板插件"：将画板③（设定）扩展为"案件线索板"，新增"嫌疑人时间线"和"证据链"两个子面板，并调整了画板④（细纲）的输出格式以适配推理小说的"红鲱鱼/伏笔/揭露"节奏。她通过插件机制注册后，其他织梦机用户可以在项目设置中选择使用"推理画板"替代默认画板。

**触发时刻：** 用户创建新项目时选择"推理小说"模板 → 自动加载推理画板插件。
**价值主张：** 织梦机的方法论管线从内置的文学天演论扩展为开放的创作方法论生态。

### 场景 B：AI技能开发者注册新诊断工具

[推测] 一位自由开发者熟悉织梦机的 40 个 Craft Skills 体系，认为还缺一个"对话节奏诊断"技能。他参考 CS-08（信息节奏）的 JSON Schema，编写了一个新的技能定义文件（含 prompt template、诊断维度、评分规则），通过插件系统注册到织梦机的 Skill Registry 中。用户在使用 Inspector Agent 时，可以看到新增的"对话节奏"诊断项。

**触发时刻：** 用户在 Inspector 面板中看到新增的诊断维度。
**价值主张：** 40 个第一方技能是主干，社区贡献的技能是枝叶——织梦机的文学诊断能力可以超越创始团队的知识边界。

### 场景 C：普通用户安装导出插件

[推测] 一位用户需要将织梦机的小说导出为 Scrivener 格式（.scrivx）以便在 Scrivener 中继续编辑。社区中已有开发者编写了 Scrivener 导出插件，用户一键安装后，导出菜单中新增了"Scrivener 格式"选项。

**触发时刻：** 用户完成小说 → 点击导出 → 看到新增的导出格式。
**价值主张：** 织梦机的默认导出格式（Markdown/纯文本）无法覆盖所有用户的后续工具链需求，插件生态可以低成本覆盖长尾。

### 场景 D：主题作者定制 UI

[推测] 一位 UI 设计师制作了"赛博朋克主题"插件，改变了织梦机的配色方案、字体、图标集，并在插件市场中发布。其他使用织梦机创作赛博朋克小说的用户可以安装该主题，在写作时获得沉浸式的视觉体验。

**触发时刻：** 用户在设置 > 主题中浏览社区主题 → 一键应用。
**价值主张：** 主题插件是轻量、低风险、高感知价值的插件形式，适合作为插件系统的起点。

### 场景 E：数据源插件导入外部世界设定

[推测] 一位用户已在 World Anvil 中建立了完整的世界观设定。社区有一个"World Anvil 导入插件"，可以将 World Anvil 的 JSON 导出数据映射为织梦机画板③的数据结构。用户安装插件后，通过"导入 > 从 World Anvil"完成一键迁移。

**触发时刻：** 用户从外部工具迁移到织梦机 → 需要数据导入。
**价值主张：** 降低用户的迁移成本，消除"已经在其他工具中建好设定怎么办"的转换阻力。

## 3. 可能的插件形态

以下按"从易到难"排序，每种形态附一个具体例子。

### 形态 1：导出/导入插件

| 维度 | 说明 |
|------|------|
| **定义** | 注册新的导出格式或导入格式，扩展织梦机的数据边界 |
| **复杂度** | ★☆☆☆☆（低） |
| **插件内容** | 格式转换逻辑（纯数据映射，无 UI 变更，无 AI 调用） |
| **例子** | `.docx` 导出插件：将画板⑤正文 + 画板④章节包转换为 Word 文档格式 |
| **是否影响核心管线** | 否。只在导出/导入时激活 |
| **运行时环境** | [推测] TypeScript 函数 + Rust 格式转换。导出导入不涉及 UI 变更 |
| **与现有体系关系** | v2.0.1 已有 Markdown 导出。插件可在此之上扩展新格式 |
| **技术前提** | 织梦机内部数据结构（Contract Chain）必须稳定且文档化 |

### 形态 2：主题/UI 插件

| 维度 | 说明 |
|------|------|
| **定义** | 修改织梦机的外观（配色、字体、图标、布局微调），不改变功能行为 |
| **复杂度** | ★★☆☆☆（低-中） |
| **插件内容** | CSS 变量覆盖 + 字体包 + 图标集合。参考 Obsidian 主题机制 |
| **例子** | "赛博朋克主题"：暗色背景 + 霓虹色高亮 + 等宽字体 |
| **是否影响核心管线** | 否。纯视觉层覆盖 |
| **运行时环境** | [推测] CSS 变量覆盖，无执行代码需求。如果支持动态主题，需要主题 CSS 加载机制 |
| **与现有体系关系** | 织梦机已有 1.3 DESIGN-TOKENS（CSS 变量体系），主题插件天然契合 |
| **技术前提** | DESIGN-TOKENS 的 CSS 变量覆盖机制需要文档化并稳定为公共 API |

### 形态 3：AI Skill 插件

| 维度 | 说明 |
|------|------|
| **定义** | 注册新的 Craft Skill，扩展织梦机的 AI 诊断或生成能力 |
| **复杂度** | ★★★☆☆（中） |
| **插件内容** | 遵循 `craft-skill-schema.json` 的 JSON 定义文件（含 prompt template、诊断维度、评分规则、输入输出 schema） |
| **例子** | "对话节奏诊断"：分析小说中对话段落的信息密度、节奏变化、角色声口区分度 |
| **是否影响核心管线** | 部分影响。注册的 skill 可被 Agent 层调用，但不会改变默认 Agent 链路 |
| **运行时环境** | [推测] 纯 JSON 配置，不执行自定义代码。Skill Executor 统一执行。插件只需提供 JSON 定义 + prompt template |
| **与现有体系关系** | **这是 v2.0.2 Prompt/Skill Registry 的自然延伸**——Registry 目前仅注册第一方 skill，插件化意味着 Registry 开放给第三方。这是当前架构中最"离插件最近"的形态 |
| **技术前提** | ① craft-skill-schema.json 冻结为稳定版本 ② Skill Registry 支持隔离第三方注册 ③ prompt 注入安全性检查（防 prompt injection） |

[已知事实] v2.0.2 PRD 定义了 Prompt/Skill Registry 作为织梦机 AI 能力底座的一部分，包含 5 个已注册方法论技能的 prompt 仓库。当前的 Registry 是第一方系统。

[推测] 将 Registry 从第一方扩展为第三方可注册，是织梦机从"封闭技能体系"走向"插件生态"的最小技术步长。

### 形态 4：方法论插件（画板扩展）

| 维度 | 说明 |
|------|------|
| **定义** | 注册新的画板流程、子面板、或方法论步骤，扩展或替换织梦机的默认创作管线 |
| **复杂度** | ★★★★☆（高） |
| **插件内容** | ① 画板数据结构定义（扩展或替换默认的画板数据模型）② UI 组件（React）③ AI prompt 模板 ④ 画板间数据依赖关系声明 |
| **例子** | "推理小说画板"：新增"线索板"子面板 + 修改细纲输出格式为"红鲱鱼/伏笔/揭露"三段 |
| **是否影响核心管线** | **是。** 方法论插件直接修改画板管线的行为。这是最强但也最危险的插件形态 |
| **运行时环境** | [推测] 需要 React 组件加载机制 + 数据模型注册 + 画板路由重写。技术挑战包括：UI 沙箱、数据模型兼容性、版本冲突 |
| **与现有体系关系** | 织梦机的核心假设是"五画板管线是数据强依赖链"。方法论插件打破了这个假设——它允许第三方重新定义管线拓扑。这意味着：① 插件的画板间数据流需要解决"上游产出如何映射到下游输入"的问题 ② 插件的 AI 能力需要调用底层 BYOK 层 |
| **技术前提** | ① 画板框架需要从"固定管线"重构为"可插拔管线" ② 需要定义画板间契约标准 ③ UI 组件沙箱 ④ 数据迁移/兼容方案 |

### 形态 5：数据源插件

| 维度 | 说明 |
|------|------|
| **定义** | 从外部工具/服务导入数据并映射为织梦机的内部数据结构 |
| **复杂度** | ★★★☆☆（中） |
| **插件内容** | 数据读取逻辑 + 字段映射规则 + 冲突处理策略 |
| **例子** | "World Anvil 导入插件"：读取 World Anvil 的 JSON 导出，将角色/地点/组织映射到画板③的数据模型 |
| **是否影响核心管线** | 否。只在导入阶段激活，导入完成后即完成使命 |
| **运行时环境** | [推测] TypeScript 函数，纯数据转换，无 UI 变更 |
| **与现有体系关系** | v2.3 反向管道已有"导入与切章"功能。数据源插件可以视为反向管道的扩展——新的导入来源 |
| **技术前提** | 织梦机内部数据结构（Contract Chain）稳定且文档化 |

### 形态对比总结

| 形态 | 复杂度 | 核心挑战 | 用户价值 | 建议优先级 |
|------|--------|---------|---------|-----------|
| 导出/导入插件 | ★☆☆☆☆ | Contract 稳定 | 中（覆盖长尾格式） | 第 1 批 |
| 主题插件 | ★★☆☆☆ | DESIGN-TOKENS 冻结 | 中（沉浸式体验） | 第 1 批 |
| AI Skill 插件 | ★★★☆☆ | Skill Schema 冻结 + 安全 | 高（扩展诊断能力） | 第 2 批 |
| 数据源插件 | ★★★☆☆ | Contract 稳定 | 中（降低迁移门槛） | 第 2 批 |
| 方法论插件 | ★★★★☆ | 画板架构重构 | 最高（定义新创作范式） | 第 3 批（最晚期） |

## 4. 技术可行性初步判断

### 4.1 架构现状

[已知事实] 织梦机的当前技术栈：

```
Tauri v2 (Rust + React + TypeScript)
├── Rust 后端: BYOK 层 / SQLite / Tauri commands
├── TypeScript 前端: Agent 层 / Skill 层 / UI 组件
│   ├── Skill Layer: Skill Registry (JSON) + Executor + Chain Engine
│   ├── Agent Layer: 状态机路由 + 深度控制
│   └── UI Layer: React + TipTap + ReactFlow
└── 跨层通信: Tauri invoke / events
```

### 4.2 各形态的技术可行性

#### 形态 1（导出/导入插件）：**可行**

[推测] TypeScript 函数 + Rust 格式转换的架构是成熟的。具体方案可以是：
- 插件是一个 npm 包（纯 TypeScript 函数，转换数据），或一个 JSON 配置（注册导出格式名称 + 转换函数路径）
- 织梦机在导出/导入时扫描已安装插件，动态加载格式
- 核心挑战不是技术，而是 Contract Chain 的稳定性和文档化——如果内部数据结构频繁变动，所有导出/导入插件都会断

#### 形态 2（主题插件）：**可行**

[推测] 织梦机已有 1.3 DESIGN-TOKENS，所有 CSS 变量有统一入口。主题插件只需覆盖这些变量：
- 插件格式：`theme.json`（配置文件，含 CSS 变量覆盖值 + 字体声明 + 元数据）
- 安装机制：将主题文件复制到 `~/.zhimengji/themes/` 目录
- 应用机制：在 `:root` 选择器上应用变量覆盖
- 核心挑战：确保 DESIGN-TOKENS 覆盖所有 UI 组件

#### 形态 3（AI Skill 插件）：**可行，但有前提条件**

[推测] 这是当前最接近"可插件化"的形态，因为：
- 已有完整的 `craft-skill-schema.json` JSON Schema
- 已有 Skill Registry 从 JSON 加载 skill 定义的机制
- 已有 Skill Executor 统一执行机制

需要解决的前提条件：

| 条件 | 当前状态 | 需要做的 |
|------|---------|---------|
| Skill Schema 冻结 | v2.0.2 仍在建设中 | 在 v2.0.2 完成后冻结 Schema 版本，建立向后兼容策略 |
| Registry 隔离 | 当前为第一方注册 | 新增"第三方注册区"，与第一方 skill 隔离，避免覆盖或冲突 |
| Prompt 安全性 | 未评估 | 第三方 prompt template 可能包含注入攻击。需要 prompt 预检机制 |
| 模型路由限制 | 可复用现有模型路由 | 第三方 skill 可能需要限制使用用户的低配模型（如 Haiku 而非 Opus） |
| 名称空间冲突 | 未定义 | 第三方 skill ID 应加 namespace 前缀（如 `community.author_name.skill_id`） |

#### 形态 4（方法论插件）：**技术不可行，需要架构重构**

[推测] 当前画板架构的约束使方法论插件在当前阶段不可行：

| 约束 | 当前实现 | 插件需求 | 差距 |
|------|---------|---------|------|
| 画板拓扑 | 固定五画板串行链 | 可扩展/替换画板 | 需要"可插拔画板框架" |
| 数据模型 | 每个画板固定数据结构 | 可扩展/替换数据模型 | 需要"数据模型注册表" |
| UI 组件 | 每个画板渲染固定组件 | 可加载第三方组件 | 需要"UI 组件沙箱" |
| 画板间依赖 | 强编码的上下游关系 | 动态声明依赖关系 | 需要"契约化画板间 API" |

[推测] 方法论插件的技术架构至少需要：
1. **画板抽象层**（Canvas Abstraction Layer）——定义画板的通用接口（输入/输出/验证/渲染）
2. **组件沙箱**——第三方 React 组件的隔离执行环境（iframe + postMessage 或 web workers）
3. **数据模型注册表**——第三方画板的数据结构与现有数据的兼容映射
4. **依赖解析器**——画板间数据流动的拓扑排序和验证

以上每一项都是"重构级"的工作量[推测]，不应在当前春生阶段（v2.0-H ~ v2.4）启动。

#### 形态 5（数据源插件）：**可行**

[推测] 与导出/导入插件类似，TypeScript 函数映射的数据转换场景。接入 v2.3 的反向管道导入机制。

### 4.3 整体技术可行性判断

```
形态 1（导出/导入）──── 可行 ──── 当前可做
形态 2（主题）──────── 可行 ──── 当前可做
形态 3（AI Skill）──── 有条件可行 ── v2.0.2 完成后可启动
形态 4（方法论）────── 不可行 ── 需要架构重构，至少 v2.6 后
形态 5（数据源）────── 可行 ──── 可接入 v2.3 反向管道
```

### 4.4 插件 API 的最小集合（初步推测）

[推测] 一个最小可行的插件 API 可能包含以下部分：

```
Plugin API v0.1（推测的最小集合）
├── Registry API
│   ├── registerSkill(skillDef: SkillDefinition) → void
│   ├── registerExportFormat(name: string, handler: ExportHandler) → void
│   └── registerImportFormat(name: string, handler: ImportHandler) → void
├── Data API（只读）
│   ├── getCurrentProject() → Project
│   ├── getCanvasData(canvasId: string) → CanvasData
│   └── getWorldObjects(filter?: WorldObjectFilter) → WorldObject[]
├── Settings API
│   ├── saveSetting(key: string, value: any) → void
│   └── getSetting(key: string) → any
└── UI API（限主题插件）
    ├── overrideCSS(theme: ThemeConfig) → void
    └── registerFont(name: string, url: string) → void
```

[推测] 特别注意：插件 API 的**设计过程本身就有巨大成本**——定义 API 即定义契约。一旦发布，任何 breaking change 都会破坏生态。因此插件 API 应在织梦机内部 Contract Chain 稳定后才开始设计。

## 5. 与现有体系的映射关系

### 5.1 Prompt/Skill Registry（v2.0.2）——插件的前身？

[已知事实] v2.0.2 的 Prompt/Skill Registry 已注册以下技能：
- `premise.five_step`（画板①五步最小方法论）
- `structure.l1_l4`（画板②四层结构）
- `setting.sparrow_9_3`（画板③麻雀 9+3 模式）
- `packet.three_detail_modes`（画板④三档细度）
- `draft.chapter_writer`（画板⑤正文撰写）

| 维度 | 当前 Registry | 插件化后的 Registry |
|------|--------------|-------------------|
| **注册主体** | 织梦机核心代码 | 第三方开发者 / 社区 |
| **注册方式** | 硬编码 JSON 文件 | 插件安装 → Registry 自动注册 |
| **命名空间** | 无前缀（第一方） | 社区名前缀（如 `community.author.skill_id`） |
| **审核机制** | 无（第一方代码经过代码审核） | 需要安全审核和 Schema 验证 |
| **依赖管理** | 无（第一方 skill 相互独立） | 需要声明依赖关系和版本兼容性 |
| **卸载** | 不存在（不可卸载） | 可卸载，需清理注册数据 |

[推测] **Prompt/Skill Registry 是插件系统的最佳切入点**。如果未来要做插件，从 "Registry 开放给第三方" 开始，比从头造一个插件框架成本低得多。

### 5.2 Craft Skill 体系（40 个 CS）——技能插件的基底

[已知事实] 织梦机的 40 个 Craft Skills 是基于文学天演论六部理论的诊断/生成技能。每个 skill 以 JSON 定义，包含：
- promptTemplate（system + user prompt）
- lens（诊断维度 + 权重）
- archetype（角色类型）
- depthControl（深度控制配置）
- boundary（适用范围限制）

[推测] 如果第三方要注册新的 AI Skill，他们需要：
1. 理解 `craft-skill-schema.json` Schema（需文档化并冻结）
2. 编写符合 Schema 的 JSON 定义（含有效的 prompt template）
3. 声明自己的 skill 是独立技能还是对现有技能的增强
4. 通过 Schema 验证和安全检查

**Skill 插件与第一方 skill 的差异：**

| 维度 | 第一方 Skill（CS-01 ~ CS-40） | 第三方 Skill 插件 |
|------|-------------------------------|-------------------|
| 可用 Agent | 接入完整 Agent 链路（Architect/Writer/Reader/Inspector） | 仅作为独立诊断可选，不进入默认 Agent 链 |
| 调用方 | Agent 自动编排 | 用户手动触发，或通过自定义编排 |
| 优先级 | 默认启用 | 安装后需用户手动启用 |
| 深度控制 | 三路深度（auto/suggest/manual）全支持 | 限制为 suggest 级别（需用户确认） |

### 5.3 画板框架（五画板管线）——方法论插件的对接口

[已知事实] 织梦机的五画板管线由以下组成：
- 画板① → 画板② → 画板③ → 画板④ → 画板⑤
- 每个画板有独立的数据结构
- 画板间为强数据依赖链

[推测] 若要支持方法论插件（替换画板或插入子面板），织梦机需要暴露：
1. **画板接口定义**——每个画板的输入/输出 Schema
2. **画板注册表**——画板类型与 UI 组件的映射
3. **管线配置**——用户可在项目设置中选择"推理画板"替代默认画板

**重要判断：** 方法论插件对当前织梦机的改动程度接近"重新设计画板架构"[推测]，不应在当前阶段考虑。

### 5.4 Plugin 和 Skill 的关系

[推测] 明确了以下层级关系：

```
Plugin（广义）
├── AI Skill Plugin  ── 扩展 Skill Registry（最轻量，最自然）
├── Export/Import Plugin ── 扩展数据边界
├── Theme Plugin     ── 扩展视觉层
├── Data Source Plugin ── 扩展反向管道
└── Methodology Plugin ── 扩展画板管线（最重量）
       └── 内含多个 AI Skills + 画板 UI + 数据模型

Skill（狭义） ── 一个 AI 能力单元
├── 由 promptTemplate + lens + archetype 组成
└── 在 Agent 编排链中作为执行节点
```

**关键结论：** Skill 是插件的"基础能力单元"。一个方法论插件可以包含多个 Skill，但一个 Skill 插件只注册一个 AI 能力。两者关系类似于"程序"和"函数"。

### 5.5 现有"不做清单"中的插件决策

[已知事实] v0.3.1 PRD 第 21.1 节 "已决定不做的功能清单" 中明确列出：

> **插件/扩展系统** → 不做。理由：方法论壁垒要求保持核心体验的一致性。替代方案：通过模板市场扩展输出格式。

[已知事实] v2.6 Out of Scope 清单也包含：
> 插件/扩展系统

[推测] 上述决策的背景：
1. 时间维度：当前（春生阶段）最紧迫的是产品价值验证（PMF），不是平台化扩展
2. 战略维度：方法论壁垒要求织梦机的核心体验（五画板管线 + 文学天演论技能）由第一方把控
3. 资源维度：插件系统的基础设施成本（API 设计、沙箱、市场、审核）在当前阶段资源不足

## 6. 建议的时间窗口

### 6.1 前置条件

[建议] 插件系统启动前必须就绪的条件：

| 前置条件 | 对应版本 | 说明 |
|---------|---------|------|
| **Core Contract Chain 稳定** | v2.1.1 之后 | 画板间的数据契约必须文档化、版本化、向后兼容。如果内部数据结构还在频繁变动，任何插件都会被波及 |
| **Skill Schema 冻结** | v2.0.2 之后 | `craft-skill-schema.json` 必须有稳定版本，且第一方 40 个 skill 全部就绪 |
| **DESIGN-TOKENS 覆盖全部 UI** | v2.0 之后逐步 | 主题插件依赖 CSS 变量的完整覆盖 |
| **产品价值验证通过** | v2.1.2 Go 决策之后 | 如果织梦机的产品价值（方法论管线）未被验证，插件系统不能拯救产品方向 |
| **资源充裕** | v2.4 商业化稳定后 | 插件系统基础设施需要持续投入，商业化后才能支撑 |

### 6.2 建议的时间窗口

```
版本路线        插件窗口
─────────────────────────────────────
v2.0-H          不可开始
v2.0.1          不可开始
v2.0.2          ── 建立 Prompt/Skill Registry（第一方）→ 这是插件的基础设施
v2.1.0          不可开始
v2.1.1          不可开始
v2.1.2          不可开始（价值验证关键）
v2.2            ── 可以考虑启动"轻量插件"探索（主题 + 导出格式）
v2.3            ── 数据源插件可接入反向管道
v2.4            ── 商业化就绪后，可评估插件系统的资源投入
v2.5            ── 模板市场可以作为插件系统的试验田
v2.6            战略分叉点：如果要平台化，插件系统是必要条件
```

### 6.3 建议的启动策略

[建议] 不要一次性启动"插件系统"。分三步渐进：

**第一步（v2.2 ~ v2.4）：零代码插件**
- 仅支持 JSON 配置型插件（主题插件 = `theme.json`，Skill 插件 = 符合 Schema 的 JSON）
- 不支持任何自定义代码执行
- 插件通过本地目录安装（`~/.zhimengji/plugins/`）
- 核心价值：验证"插件安装/启用/卸载"的用户体验流程

**第二步（v2.5 ~ v2.6）：轻量代码插件**
- 支持 TypeScript 函数的导出/导入插件
- 插件在 web worker 沙箱中执行（无 DOM 访问、无文件系统访问）
- 引入插件市场概念（浏览、安装、评分）
- 核心价值：验证第三方开发者社区的参与意愿

**第三步（v2.6+）：全量插件框架**
- 支持 UI 组件加载（方法论插件）
- 完善插件 API
- 插件版本管理 / 兼容性检测 / 自动更新
- 引入安全审核机制
- 核心价值：成熟开放的插件生态

### 6.4 关于 v2.6 策略分叉的说明

[已知事实] PRD 明确将 v2.6 标记为"战略高风险 / 非本地产品分叉点"，并且"插件/扩展系统"当前在不做清单中。

[建议] **即使不启动完整插件系统，织梦机也应为未来可能的插件化做好架构准备：**

1. **Contract Chain 文档化**——无论插件做不做，稳定的契约文档对维护和调试都有利
2. **Skill Schema 冻结**——Skill Registry 是织梦机 AI 能力的核心，冻结后更方便迭代
3. **DESIGN-TOKENS 覆盖**——CSS 变量体系对团队协作和 UI 一致性有直接帮助
4. **模块内部解耦**——画板框架的模块边界清晰，不仅有利于插件化，也有利于单体维护

以上 4 项准备工作几乎没有"不做的理由"[建议]，无论插件最终做不做。

## 7. 开放问题

以下问题待后续讨论，当前阶段不需要结论。

### Q1：插件系统的第一用户是谁？

插件系统的用户有两类——插件开发者和插件使用者。两者的需求存在潜在冲突：
- 插件开发者需要：灵活的 API、完善的文档、调试工具、发布渠道
- 插件使用者需要：一键安装、安全可靠、不降低性能、不破坏现有功能

[推测] 对于织梦机的春生阶段，插件系统的"第一用户"不应是外部开发者，而应是织梦机自己的方法论团队。插件系统先作为"内部扩展机制"运行，成熟后再开放给第三方。

### Q2：插件系统是否与"方法论壁垒"战略冲突？

[已知事实] 织梦机的竞争壁垒之一是"自有方法论"——文学天演论六部理论不可复制。

[推测] 开放方法论插件存在一个战略风险：如果第三方方法论插件被证明比织梦机的默认方法论更好用，织梦机的核心壁垒被相对稀释。另一个可能性：如果插件系统被竞品（如 Novelcrafter）采用并做出更开放的方法论体系，织梦机可能陷入"封闭 vs 开放"的竞争劣势。

[推测] 两种可能的方向：
- **保守策略**：方法论插件严格受控，由织梦机官方审核和发布。第三方可以贡献但需要合并到官方体系
- **开放策略**：完全开放方法论注册，织梦机成为"创作方法论平台"。此时织梦机的定位从内容工具变为方法论基础设施

这两个策略对应完全不同的产品定位和商业模式。

### Q3：插件如何保障用户数据安全？

[推测] 第三方插件（尤其是包含代码执行的插件）可能：
- 读取 / 泄露用户的写作内容
- 读取 / 泄露用户的 API Key
- 在用户机器上执行恶意操作
- 破坏项目数据

安全问题需要考虑的维度：
- AI Skill 插件：prompt injection 风险（第三方的 prompt template 可能引导 LLM 输出有害内容）
- 导出插件：文件系统访问权限需要受限
- 方法论插件：数据模型变更可能导致数据损坏

### Q4：插件的商业模式是什么？

[推测] 可能的模式：
- **免费插件市场**：插件免费，吸引用户，插件开发者的回报来自社区声望
- **付费插件市场**：插件开发者定价，织梦机抽成（类似 WordPress 插件市场）
- **插件作为获客工具**：免费插件吸引用户使用织梦机，用户为织梦机核心功能付费
- **企业插件授权**：大型方法论体系（如"好莱坞编剧模板"）由机构制作并授权
- **闭环生态**：插件是织梦机的护城河，不直接盈利但提升用户黏性

不同的商业模式需要不同的基础设施投入（支付系统、审核团队、客服支持）。

### Q5：插件与模板市场（v2.5）的关系？

[已知事实] v2.5 的模板市场是"将织梦机管线产出的内容以不同形式呈现"（漫剧/短剧模板）。

[推测] 模板和插件是两个不同的概念：
- **模板**：输出格式预设（漫剧模板、网文模板），用户选择后管线的产出形态发生变化
- **插件**：扩展织梦机能力（新画板、新技能、新格式），安装后织梦机本身的能力边界扩大

但两者也存在交集——一个"推理小说插件"可能既包含画板扩展（插件部分），又包含输出格式预设（模板部分）。建议未来在 v2.5 模板市场的基础上评估是否需要将两者合并或分离。

### Q6：插件是否需要版本管理和依赖解析？

[推测] 如果插件数量超过 20 个，以下问题必然出现：
- 插件 A 依赖插件 B 的某个能力
- 插件 A 的 v1.0 在织梦机 v2.4 上可用，但 v2.0 需要织梦机 v2.6
- 两个插件同时修改了同一个设置或技能

[推测] 版本管理是"不做没人抱怨，做了没人看见"的基础设施工作——它应该在前 20 个插件之前就设计好，而不是之后。

## 8. 不做判断

### 当前明确不做什么

基于织梦机当前的春生阶段（产品价值验证期）和资源约束，以下事项**明确不在前期插件探索范围内**：

#### 8.1 不启动插件系统开发

[建议] 在当前阶段（v2.0-H ~ v2.1.2），插件系统的开发不应被启动。织梦机的核心任务是：
1. 主流程可用（v2.0-H）
2. 操作阻力验证（v2.0.1）
3. AI 能力底座（v2.0.2）
4. 方法论骨架（v2.1.0）
5. 方法论集成（v2.1.1）
6. 产品价值验证（v2.1.2）

插件系统是"产品价值被验证后"才需要考虑的事项。

#### 8.2 不做完整的画板重构

[建议] 方法论插件所需画板框架重构（Canvas Abstraction Layer + 组件沙箱）不应在 v2.4 之前启动。画板架构在当前阶段应保持稳定，以支持方法论骨架（v2.1.0）的交付。

#### 8.3 不做插件市场/商店

[建议] 插件市场涉及支付、审核、托管、客服等全套平台化基础设施。在织梦机完成商业化验证（v2.4）之前，插件市场不应启动。

#### 8.4 不做代码执行沙箱

[建议] 第三方代码执行沙箱（用于方法论插件的 React 组件、数据转换脚本等）需要深入的技术投入。在插件系统的"零代码 JSON 配置阶段"验证通过之前，不启动沙箱开发。

#### 8.5 不做插件 API 设计

[建议] API 设计应该在织梦机内部 Contract Chain 稳定之后再做。现在的 API 设计是"在沙子上盖房子"——Contract 都没稳定，API 设计出来也会很快过时。

### 但可以做什么（零成本准备）

[建议] 以下准备工作几乎零成本，且对插件化和非插件化都有益：

| 准备项 | 说明 | 受益方（插件化后） | 受益方（不插件化） |
|--------|------|------------------|-------------------|
| **Contract Chain 文档化** | 写清楚每个画板的输入/输出 Schema | 插件开发者 | 团队内部沟通 |
| **Skill Schema 冻结** | 发布 v1.0 版 craft-skill-schema.json | 第三方 Skill 开发者 | AI 能力稳定性 |
| **DESIGN-TOKENS 覆盖审计** | 检查是否所有组件都用了 CSS 变量 | 主题插件开发者 | UI 一致性 |
| **模块边界画清楚** | 在架构文档中标记哪些模块是"公共 API"，哪些是"内部实现" | 插件框架开发者 | 维护成本降低 |
| **安装目录规划** | 决定插件的本地文件系统位置（如 `~/.zhimengji/plugins/`） | 插件安装机制 | 数据组织清晰 |
| **命名空间策略** | 定义插件和 skill 的命名规则（如 `community.author_name.plugin_name`） | 插件冲突避免 | 代码组织规范 |

以上准备项**即使最终不做插件系统，也是好的工程实践**[建议]。

## 附录：参考产品分析（简要）

### Novelcrafter 的"插件"形态

[推测] 基于公开资料分析的结论：

Novelcrafter 没有一个传统意义上的"插件系统"（没有 VS Code 式的 Extension API）。它的扩展能力主要通过以下方式实现：

| 扩展方式 | 说明 | 与织梦机的关系 |
|---------|------|-------------|
| **Codex** | 用户自定义知识条目，可以被 AI 读取 | 类似织梦机的画板③设定，但更自由 |
| **AI Agent** | Novelcrafter 近期推出的 Agent 功能，允许自定义 AI 行为 | 类似织梦机的 Agent 层 + Skill 体系，但弱于织梦机的结构化技能 |
| **自定义 prompt** | 用户在编辑器中自定义 AI 的 behavior prompt | 织梦机 v2.0.2 的 Prompt/Skill Registry 有类似功能 |
| **导出模板** | 用户创建自定义导出格式 | 对应织梦机形态 1 导出插件 |

[推测] Novelcrafter 的"扩展"方式体现了 AI 写作工具的通用趋势：从固定功能 → 可配置 prompt → 可编排 agent。但 Novelcrafter 尚未走到"第三方开发者可注册新能力"的完整插件系统。

### Obsidian 插件生态的关键启示

[推测] 基于公开资料分析的结论：

Obsidian 的插件系统是"小型独立工具依赖一个巨型平台"的典型，其成功要素包括：

| 要素 | Obsidian 做法 | 对织梦机的启示 |
|------|-------------|---------------|
| **API 稳定性** | Obsidian 保持内部 API 兼容性承诺，插件破坏率低 | 插件 API 一旦发布，breaking change 成本极高 |
| **社区审核** | 社区插件通过 GitHub 发布，Obsidian 不审核但社区有质量筛选 | 织梦机的 AI Skill 插件需要 prompt 安全审核，不能完全放权 |
| **核心 vs 外围** | Obsidian 核心功能（笔记/链接/图谱）不变，外围功能（日历/Kanban/表格）由插件实现 | 织梦机的"核心"五画板管线也应锁定，插件只能扩展外围 |
| **无限长尾** | 插件解决小众需求（数学笔记、Zettelkasten、每日记录），Obsidian 专注通用体验 | 导出/导入插件是最典型的长尾场景 |

[推测] Obsidian 的插件系统用了约 2 年（从 v0.0.1 到 v0.8.0 引入社区插件）才成熟。织梦机插件系统的节奏可参考：先稳定核心，再开放外围。

### VS Code 插件生态的启示

[推测] VS Code 的插件系统是"平台型"的极致，但它面向的是开发者（本身就懂代码的人群）。织梦机的用户是非技术出身的创作者，其插件系统的复杂度和抽象层次应该更低：

| 维度 | VS Code | 织梦机（建议） |
|------|---------|--------------|
| 插件语言 | TypeScript 完整 API | JSON 配置 + 受限的 TypeScript 函数 |
| 沙箱 | Extension Host 进程隔离 | Web Worker + 无 DOM 访问 |
| API 范围 | 编辑器全部功能 | 仅限于注册能力/格式/主题 |
| 目标用户 | 开发者（懂编码） | 创作者、方法论作者（非技术） |
| 安装方式 | 市场 → 点击安装 | 市场 / 文件拖拽 |

### 9.1 当前状态：纯本地桌面应用

[已知事实] 织梦机当前的部署形态：

```
┌────────────────────────────────────────────┐
│  Tauri v2 Desktop App（用户本地）            │
│                                              │
│  ┌────────────┐  ┌──────────────────────┐   │
│  │ React UI   │  │ Rust Backend          │   │
│  │ (TS/React) │  │ BYOK / SQLite / Tauri │   │
│  │ TipTap     │  │ commands              │   │
│  │ ReactFlow  │  │                       │   │
│  └────────────┘  └──────────┬───────────┘   │
│                             │                │
│                   LLM API ←─┘ (用户自带 Key) │
│                   调用直接走出站 HTTP          │
└────────────────────────────────────────────┘
```

| 特征 | 说明 |
|------|------|
| **数据存储** | 全部在本地 SQLite，用户机器上的 `~/.zhimengji/` 目录 |
| **AI 调用** | 用户自带 API Key（BYOK），织梦机不出服务器，直连 LLM Provider |
| **网络依赖** | 仅 AI 调用需要出站 HTTP。纯离线时核心创作管线可手动编辑 |
| **用户账户** | 无（不需要——数据全在本地） |
| **多设备** | 不支持（数据锁在单机） |
| **协作** | 不支持 |
| **版本** | v2.0 Beta Skeleton Complete |

### 9.2 织梦机需要服务器吗？

[推测] 这个问题的答案取决于织梦机的产品定位演进的三个阶段。关键判断是：**服务器不是织梦机的"必经之路"，而是"平台化的必要条件"**。

#### 阶段一：纯本地工具（当前 ~ v2.4 商业化前）

| 维度 | 判断 |
|------|------|
| **需要服务器？** | **完全不需要。** 核心价值（方法论管线 + AI 辅助写作）全部可在本地实现 |
| **服务器能带来什么？** | 云备份、多设备同步——这些是"更好用"而非"必须" |
| **服务器的风险** | 运维成本、数据合规、离线不可用——对春生阶段是额外负担 |
| **建议** | 坚守纯本地。所有精力投入产品价值验证 |

[建议] 在 v2.4（商业化）之前，织梦机的所有核心能力都应该继续以纯本地形态交付。

#### 阶段二：本地 + 可选云端（v2.4 商业化后 ~ v2.6 前）

[推测] 商业化后，以下场景可能催生对服务端的有限需求：

| 场景 | 是否需要服务器？ | 优先级 |
|------|----------------|--------|
| **托管 AI 额度**（用户不自带 Key，消耗平台点数） | **是**。需要服务端验证额度、记录用量、防止盗用 | **P0**（商业化的前提） |
| **项目云备份/同步** | 可选。用户可选择开启云同步、跨设备写作 | P1 |
| **用户账户系统** | 需要。托管额度和付费订阅都需要账户 | P0（跟托管额度走） |
| **插件/模板市场** | 需要。需要服务端存储、分发、计费 | P1（v2.5 模板市场） |
| **AI 代理/中转服务** | 可选。织梦机做 LLM API 的中转代理，用户不需自带 Key | 这是"托管额度"的同一种能力 |

[建议] v2.4 商业化阶段的最小服务器形态是：**仅做额度管理 + 用户账户**。不做云同步，不做协作，不做插件市场。最小化服务器的范围和运维复杂度。

#### 阶段三：混合平台（v2.6 及以后）

[已知事实] PRD 将 v2.6 标记为"战略高风险 / 非本地产品分叉点"。v2.6 的核心变化包括：协作编辑、授权管理、交易系统。

[推测] v2.6 的每一个核心功能都**需要服务器**：
- 协作编辑 → 需要服务端同步冲突解决
- 授权管理 → 需要用户权限服务
- 交易系统 → 需要订单 / 支付 / 对账服务
- 多设备访问 → 需要数据同步服务

[建议] v2.6 不是一个"在本地加几个功能"的版本，而是**织梦机从纯本地应用转向混合架构的战略分叉点**。这个决定不应轻率做出，需要在 v2.4 商业化验证通过后才能评估。

---

### 9.3 可能部署形态对比

[推测] 四种可能的部署形态，按"服务器依赖程度"从低到高排列：

#### 形态 A：纯本地（当前，推荐保持至 v2.4）

```
用户机器
┌────────────────────┐
│ 织梦机 Desktop App │
│  SQLite (所有数据) │
│  BYOK AI 调用     │
│  (出站 HTTP)      │
└────────────────────┘
     │
     ▼ LLM Provider (OpenAI/Anthropic/DeepSeek...)
```

| 维度 | 评估 |
|------|------|
| **优点** | 零运维、数据完全在本地、完全离线可用（AI 调用除外）、无隐私合规问题 |
| **缺点** | 无多设备、无协作、无云备份、用户必须自带 Key |
| **适合阶段** | v2.0 ~ v2.4 商业化前 |
| **服务器需求** | 无 |

#### 形态 B：本地 + 额度服务器（v2.4 商业化形态）

```
用户机器                            织梦机云端
┌────────────────────┐       ┌──────────────────────┐
│ 织梦机 Desktop App │       │ 额度服务 (轻量)       │
│  SQLite (项目数据) │ ←───→ │  □ 用户账户验证       │
│  BYOK AI 调用     │       │  □ 点数余额查询        │
│  (出站 HTTP)      │       │  □ 用量日志记录        │
└────────────────────┘       │  □ 付费订阅管理        │
     │                       └──────────────────────┘
     ▼ LLM Provider
     (OpenAI/Anthropic/...)
```

| 维度 | 评估 |
|------|------|
| **优点** | 核心创作管线保持本地、离线可用。服务器极小（仅额度 + 账户）、运维成本低 |
| **缺点** | 仍无多设备、无协作、用户本地数据丢失无法恢复 |
| **适合阶段** | v2.4 商业化初期 |
| **服务器需求** | 轻量（1-2 台云服务器 / serverless 函数）。后端用 Rust/Node.js 均可 |

[推测] 这是从纯本地到有服务器的**最小迁移路径**。核心逻辑：用户在本地创作，与服务端仅交换"额度"和"账户"信息。项目数据完全不上传。

#### 形态 C：混合架构（v2.5 ~ v2.6 平台转型方向）

```
用户机器                            织梦机云端
┌────────────────────┐       ┌──────────────────────────┐
│ 织梦机 Desktop App │       │ 核心服务                  │
│  SQLite (本地缓存) │ ←───→ │  □ 用户账户              │
│  BYOK / 托管 AI   │       │  □ 项目同步/存储          │
│  (出站 HTTP)      │       │  □ 额度管理              │
│  (WebSocket 同步) │       │  □ 插件/模板市场          │
└────────────────────┘       │  □ 协作编辑 (v2.6+)       │
                             │  □ 公开分享               │
                             └──────────────────────────┘
                                  │
                                  ▼ LLM Provider
                                  (平台托管额度)
```

| 维度 | 评估 |
|------|------|
| **优点** | 多设备同步、协作能力、云备份、插件市场、公开分享 |
| **缺点** | 运维成本大幅上升、离线降级、数据隐私复杂度、需要团队持续维护 |
| **适合阶段** | v2.6 平台化阶段（仅当 v2.4 商业化验证通过 + v2.5 模板数据支撑） |
| **服务器需求** | 中等。至少需要：API 服务、数据库、对象存储、WebSocket 服务 |

#### 形态 D：全云端 Web 版（最远期）

| 维度 | 评估 |
|------|------|
| **说明** | 织梦机以 Web 应用形态运行，所有数据在云端。用户通过浏览器访问 |
| **适合场景** | ChatGPT 的"打开即用"模式——用户不需安装任何软件 |
| **技术挑战** | BYOK 在 Web 环境不可用（Key 暴露问题）。架构决策见 PRD v0.3.1："Web 版 BYOK 方案待 v1.7 结束时重新评估" |
| **与桌面版的关系** | [推测] 桌面版和 Web 版应该共享同一套后端服务，前端适配不同平台 |

[推测] Web 版不是织梦机当前路线的自然终点。织梦机的核心价值（画板管线、方法论引导）在桌面端表现最好。Web 版作为"轻量预览/分享"形态可能更有意义，而非"替代桌面版"。

### 9.4 服务器需要的能力（从最小到完整）

[推测] 以下按"必须的服务器能力"排序：

```
v2.4 商业化最小服务器
├── 用户账户系统（注册/登录/密码重置）
│   └── [推测] 使用第三方认证服务（Auth0 / Clerk）减少自研
├── 点数/额度管理
│   ├── 余额查询（应用启动时 + 每次 AI 调用前校验）
│   ├── 用量记录（每次 AI 调用上报 token 消耗）
│   ├── 套餐管理（免费额度 → 付费套餐切换）
│   └── 支付集成（Stripe / LemonSqueezy / 支付宝）

v2.5 模板市场（在 v2.4 服务器上扩展）
├── 模板存储（模板文件 + 预览图）
├── 模板下载/安装 API
└── 模板流行度统计（可选）

v2.6 平台化（在 v2.5 之上扩展）
├── 项目同步/存储
│   ├── 项目结构化数据同步（CRDT / 操作日志）
│   ├── 文件附件存储（章节包、导出文件）
│   └── 冲突解决策略
├── 协作编辑
│   ├── WebSocket 实时同步
│   └── 权限管理
├── 公开分享
│   ├── 分享链接生成
│   └── 阅读视图（只读 Web 版预览）
└── 交易系统
    ├── 插件市场计费
    ├── 内容交易（可选）
    └── 对账/结算
```

[推测] **关键评估：** 从 v2.4 到 v2.6，服务器复杂度从"轻量额度校验"增长到"WebSocket 实时协作平台"，这不是线性增长，而是**数量级跃升**。v2.6 的服务器研发和运维成本可能是 v2.4 的 10 倍以上。

### 9.5 与当前架构的关系

#### Tauri 2 的服务器通信能力

[已知事实] Tauri 2 的 HTTP 通信机制：
- Rust 端：`reqwest` crate（已引入用于 BYOK LLM 调用）
- TS 端：`fetch` API 或 `@tauri-apps/api/http`
- WebSocket：Tauri 2 支持 WebSocket（`tauri-plugin-websocket` 或 Rust 端 `tokio-tungstenite`）
- 身份验证：可通过 Rust 端管理 token，不暴露给 JS 层

[推测] Tauri 2 的技术栈完全支持桌面应用与服务端通信。织梦机不需要更换框架即可支持服务器部署。

#### SQLite 数据同步到云端

[已知事实] 当前所有数据存储在本地 SQLite 中（约 10 张表）。

[推测] 从本地 SQLite 到云端同步的迁移路径：
1. **最低成本方案**：不上传原始 SQLite 文件。服务端用 PostgreSQL/MySQL，通过 API 进行 CRUD 操作。本地 SQLite 保持为"离线缓存"。
2. **完整同步方案**：CRDT（Conflict-free Replicated Data Types）或操作日志（Op Log）方式，支持离线编辑后同步。这是协作编辑的基础。
3. **不推荐的方案**：直接同步整个 SQLite 文件。移动端同步可能冲突，数据库文件格式可能随版本变更。

[推测] 最小服务器版本（v2.4）不涉及数据同步。数据同步是 v2.6 的能力。

#### 现有数据库表结构的云端适配

[已知事实] 当前约 10 个 SQLite 表。

[推测] 哪些表需要上服务器：

| 表 | 是否需要云端 | 理由 |
|---|------------|------|
| `projects` | 可本地 | 纯本地工具时可完全不上传。平台化时需要同步到云端 |
| `world_objects` | 可本地 | 同上 |
| `connections` | 可本地 | 同上 |
| `canvas_states` | 可本地 | 同上 |
| `api_keys` | **坚决不上传** | API Key 必须永远存储于用户本地。云端的额度管理不应接触用户原始 Key |
| `api_usage_logs` | **需要上服务器** | 额度管理依赖用量数据 |
| `usage_quotas` | **需要上服务器** | 额度配额由服务端控制 |
| `style_profiles` | 可本地 | 风格画像是项目的一部分，同 projects |
| `agent_sessions` | 可本地 | Session 状态本地管理即可 |
| `conversation_sessions` / `conversation_messages` | 可本地 | AI 对话数据本地存储，除非需要跨设备同步 |

[已知事实] v0.3.1 PRD 中 BYOK 定价原则已明确：BYOK 用户的 AI 调用的模型成本由用户自己承担，平台只收取平台服务费（功能点/高级功能额度）。这个原则天然适合"本地 BYOK AI 调用 + 服务器额度管理"的混合架构。

### 9.6 服务器对产品形态意味着什么？

[推测] 从纯本地到有服务器，织梦机的产品形态将发生以下变化：

| 维度 | 纯本地 | 混合架构（有服务器） |
|------|--------|-------------------|
| **数据主权** | 用户完全控制 | 数据在云端，用户信任织梦机运维 |
| **离线使用** | 完整离线（AI 除外） | 部分离线（依赖缓存） |
| **安全模型** | 无账户，用户机器安全由用户负责 | 需要账户密码 / 2FA / session 管理 |
| **运维职责** | 无（织梦机不运行任何服务） | 需要 7x24 运维、故障恢复、灾备 |
| **隐私合规** | 用户数据不出机器 | 需要隐私政策、GDPR/个保法合规 |
| **产品形态** | 工具 | 平台 |
| **用户锁定** | 低（项目文件可导出、可迁移） | 中高（数据在云端，迁移成本升高） |
| **商业模式** | 应用购买 / 内购 | 订阅制 / 点数制 / 市场抽成 |
| **用户心理** | "我的工具" | "我用织梦机平台" |

[推测] 最重要的变化是第 9 项：**用户的感知从"工具"变为"平台"**。纯本地应用的用户买的是软件功能，平台用户买的是服务和生态。两者的增长模型、用户预期、竞争对手完全不同。

### 9.7 时机判断

```
版本路线        服务器化时机
────────────────────────────────────────────────────────
v2.0-H
v2.0.1          纯本地。无服务器需求
v2.0.2          纯本地。无服务器需求
v2.1.0
v2.1.1
v2.1.2          ── 价值验证关键。此时不应分心做服务器
v2.2            开始定价决策准备
v2.3
v2.4            ── 最小服务器上线（额度管理 + 用户账户）
                   这是织梦机第一次需要云服务。
v2.5            可选：模板市场服务器（在 v2.4 服务器上扩展）
v2.6            ── 平台化服务器（协作/分享/交易）
                   这是战略分叉点。
```

#### 关键决策：什么时候第一次上服务器？

[建议] **v2.4 商业化之前不上服务器。** 理由：

1. **产品价值尚未验证。** v2.1.2 的 A/B 测试可能证明织梦机方法论不具备差异化价值。在价值被验证之前，任何服务器投入都是浪费。

2. **纯本地足以验证 PMF。** BYOK 模式允许用户在纯本地环境下完成全部创作流程，包括 AI 调用。不需要服务器支持即可验证产品价值。

3. **服务器的运维复杂度分散产品焦点。** 当前团队资源应当集中在方法论管线（v2.1.0）、集成（v2.1.1）和价值验证（v2.1.2）上，而非搭建基础设施。

4. **v2.4 的最小服务器可以复用成熟方案。** 用户账户、支付集成、API 认证都有成熟的第三方服务（Auth0、Stripe、Supabase），不需要从零搭建。

#### 商业化前可以准备的零成本工作

[建议] 以下准备不需要搭建服务器，但对未来的服务器化有益：

| 准备项 | 说明 | 受益版本 |
|--------|------|---------|
| **用量日志结构化** | 确保 `api_usage_logs` 表的数据结构在未来可以直接对接到云端 | v2.4 额度管理 |
| **API 层设计为可远程化** | TS 端的 `byok-api.ts` 设计为调用后端抽象层。未来后端可以是本地 Rust 或远程 API，前端调用方式不变 | v2.4 + |
| **区分本地 vs 云端数据** | 在第一方代码中标记哪些数据可上云、哪些绝不。如 `api_keys` 表标记为 NEVER_UPLOAD | v2.4 + |
| **离线降级策略设计** | 明确各功能在离线状态下的行为。在纯本地阶段就定义好"降级规范"，未来加服务器时直接复用 | v2.6 平台化 |
| **数据导出完整性** | 确保项目数据可以完整导出为可移植格式。这既是对用户自由的保障，也是未来云端导入/导出的接口 | 所有版本 |

### 9.8 服务器部署与插件系统的关系

[推测] 插件系统和服务器部署在织梦机的演进路径上是"近亲"，但不是"必须同时出现"：

```
插件系统                      服务器
───────────────────────────────
主题/导出插件      ←→  不需要服务器（纯本地即可）
AI Skill 插件      ←→  不需要服务器（通过用户 BYOK 调用 LLM）
方法论文插件       ←→  不需要服务器（画板逻辑在本地执行）
插件市场           ←→  需要服务器（模板存储/下载/计费）
协作编辑           ←→  需要服务器（WebSocket 同步）
跨设备同步         ←→  需要服务器（数据存储）
```

[推测] **关键判断：插件系统的大部分核心功能（主题/技能/方法论）可以在纯本地运作，不需要服务器。** 服务器对插件系统的价值在于"分发和市场"，而非"运行时"。

这意味着：
- 如果织梦机决定**不做平台化**（保持纯本地工具定位），仍然可以有插件生态，只是插件需要通过文件方式安装（如 Obsidian 早期的 GitHub 安装方式）
- 如果织梦机决定**做平台化**，插件市场是自然的配套
- 插件系统的"本地运行"+"文件安装"模式可以先行，服务器化的"插件市场"可以后补

### 9.9 不做什么（服务器相关）

基于织梦机当前的春生阶段和资源约束：

#### 9.9.1 当前不搭建任何服务器

[建议] v2.1.2 价值验证通过之前，不搭建任何服务器。所有能力保持纯本地。

#### 9.9.2 不做"先搭服务器等商业化"

[建议] 不建议"提前搭建服务器基础设施等商业化"。理由：
- 基础设施的技术选型可能在正式需要时已经过时
- 运维团队的招聘和维护成本会提前消耗研发资源
- 商业化前的重心应该是产品验证和用户获取，不是基础设施优化

#### 9.9.3 不做全量项目数据同步

[建议] 即使 v2.4 启动最小服务器，也不做"全量项目数据云同步"。原因：
1. 同步引入的数据冲突和离线/在线切换逻辑非常复杂
2. 用户对"写作内容上传到云端"可能有隐私顾虑
3. 项目存储需要对象存储 + 数据库，增加服务器成本

#### 9.9.4 不做 Web 版

[已知事实] PRD 已经有充分的判断：Web 版的 BYOK 方案有架构问题，v2.0 的策略是先做桌面版。

[建议] 至少在 v2.6 之前，Web 版不应进入产品路线图。

### 9.10 开放问题（服务器相关）

#### SQ1：服务器的技术栈用什么？

[推测] 织梦机已有 Rust + TypeScript 的技术团队。服务器的后端选择：
- **Rust（Actix-web / Axum）**：与现有 BYOK 层共享代码（如 LLM 调用、加密逻辑），性能最优，但 Rust Web 开发者的招聘难度高
- **TypeScript（Node.js / Bun）**：与前端共享语言，开发效率高，招聘容易，但需要额外的 Rust ↔ Node 桥接
- **serverless（Cloudflare Workers / AWS Lambda）**：运维最轻，但长连接（WebSocket）和定时任务的支持有限

[推测] 对于 v2.4 的最小服务器（额度管理 + 用户账户），serverless 方案可能是性价比最高的选择。对 v2.6 的平台化服务器，需要更完整的后端架构。

#### SQ2：服务器如何处理 BYOK 用户的 Key 安全？

[已知事实] API Key 在 Rust 端加密存储，JS 层不接触原始 Key。

[推测] 服务器化后，BYOK 的架构需要调整：
- 方案 A：Key 仍然只在本地解密。服务器只做额度验证，AI 调用继续走本地的 Rust BYOK 层。用户项目上传时过滤掉 `api_keys` 表
- 方案 B：用户在服务端登记 Key 的加密版本，由服务端代为调用 LLM。但这对用户的信任要求更高
- 方案 C：不使用平台服务器，BYOK 用户直接调用 LLM Provider 的 API

#### SQ3：服务器化后，离线能力如何保障？

[已知事实] PRD v0.3.1 已明确离线原则：核心创作管线离线可用，AI 调用在网络中断时降级。

[推测] 不同服务器化阶段对离线能力的影响：
- **v2.4 最小服务器**：额度验证在应用启动时完成，之后可离线使用。AI 调用前验证额度，如有缓存额度信息可短时离线调用
- **v2.6 平台化**：项目同步场景下，本地 SQLite 作为离线缓存。用户离线编辑后，上线时自动同步

#### SQ4：如果商业化验证失败，服务器投入是否白费？

[推测] 是的。这就是为什么 v2.4 之前不应启动服务器建设的核心原因。商业化验证失败 → 产品方向需要重新评估 → 服务器投入是沉没成本。纯本地则没有这个问题——验证失败只是代码不做进一步开发，不涉及服务停服或数据迁移损失。

[建议] 务实的策略是：在 v2.1.2 的 Go/No-Go 决策做出之前，**零服务器投入**。

### 9.11 明确不做判断（服务器相关）

[建议] 综合以上分析，以下判断明确：

**如果织梦机保持纯本地定位（v2.4 前必须保持，v2.4 后也是默认选择）：**
- 不需要服务器
- BYOK 是唯一 AI 调用方式
- 插件纯粹通过本地文件安装
- 项目数据完全在用户控制中
- 产品形态：工具

**如果织梦机走向平台化（v2.6 的战略决策）：**
- 服务器是必要条件
- 需要从额度管理开始，逐步扩展到协作/市场/分享
- 需要对离线能力进行重新设计
- 需要引入运维和安全职能
- 产品形态：平台

**两个方向都是合理的。** 当前不需要做出选择。保持纯本地直到 v2.4 商业化验证，届时根据数据和市场反馈再做决策。
